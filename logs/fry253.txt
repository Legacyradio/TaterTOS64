Handoff Action Log - fry253 - 2026-02-21

Summary
Two confirmed bugs fixed. Both prevent reliable fault telemetry and correct serial output.

---

Fix 1: kernel_linker.ld — .rodata/.data/.bss LMA alignment

Why:
The AT() clauses for .rodata/.data/.bss used raw LOADADDR+SIZEOF with no alignment,
so the LMA had no gap where VMA alignment rounds up. .text ends at VMA offset 0x144a5
but .rodata VMA starts at 0x144c0 (32-byte alignment = 0x1b byte gap). The LMA had no
matching gap: CPU read .rodata[X+0x1b] instead of .rodata[X], shifting every string
by 27 bytes. This is why serial printed "K_PMM" / "VMM" / " PANIC: %s" when the actual
calls were to "K_START\n" / "K_GDT_IDT\n" / "K_PMM\n".

Change:
  Before:
    .rodata : AT(LOADADDR(.text) + SIZEOF(.text)) { ... }
    .data   : AT(LOADADDR(.rodata) + SIZEOF(.rodata)) { ... }
    .bss    : AT(LOADADDR(.data) + SIZEOF(.data)) { ... }

  After:
    .rodata : AT(ALIGN(LOADADDR(.text) + SIZEOF(.text), ALIGNOF(.rodata))) { ... }
    .data   : AT(ALIGN(LOADADDR(.rodata) + SIZEOF(.rodata), ALIGNOF(.data))) { ... }
    .bss    : AT(ALIGN(LOADADDR(.data) + SIZEOF(.data), ALIGNOF(.bss))) { ... }

File: kernel_linker.ld

---

Fix 2: src/boot/stack_switch.asm — save RDX before `mov dx, 0xE9` corrupts it

Why:
stack_switch_and_call receives: rdi=new_rsp, rsi=fn, rdx=arg0 (handoff ptr).
`mov dx, 0xE9` is a 16-bit write that does NOT zero-extend in 64-bit mode, so it
corrupts the lower 16 bits of RDX (handoff pointer). The subsequent `mov rdi, rdx`
then passes a corrupted pointer to after_vmm, causing garbage handoff.

Change:
  Added `mov r8, rdx` before the first `mov dx, 0xE9` to save the handoff pointer.
  Changed `mov rdi, rdx` to `mov rdi, r8` to pass the preserved value.
  r8 is caller-saved in SysV AMD64 ABI — safe to use freely here.

File: src/boot/stack_switch.asm
