Summary
Fixed double vmm_phys_to_virt conversion in all four ACPI parsers.

---

Observation from fry254 run:

- after_vmm runs: 'S', 'P', kprint_init done, "TaterTOS64v3 starting" printed

- !EXC vec=0x0D RIP=0xffffffff80004cf4 (madt_init+0x64: `mov 0x4(%rax), %esi`)
- RAX=0xffff00007f778000 — NON-CANONICAL

Root Cause: Double vmm_phys_to_virt conversion
- tables.c registry_add stores physmap VIRTUAL addresses into table_registry[]:
    entries[i] → vmm_phys_to_virt(entries[i]) → physmap VMA stored in registry
- acpi_find_table returns the physmap VMA from registry (e.g. 0xffff80007f778000)
- madt_init (and fadt/mcfg/hpet_tbl) then call vmm_phys_to_virt AGAIN on that VMA:
    vmm_phys_to_virt(0xffff80007f778000)
    = 0xffff800000000000 + 0xffff80007f778000
    = 0x1ffff00007f778000 (64-bit truncated: 0xffff00007f778000)
    → NON-CANONICAL → #GP on first memory access

Fix
Remove the redundant vmm_phys_to_virt call — acpi_find_table already returns a
usable virtual address. Just cast h directly.

Files changed:
- src/kernel/acpi/madt.c line 77:
    Before: struct acpi_madt *m = (struct acpi_madt *)(uintptr_t)vmm_phys_to_virt((uint64_t)(uintptr_t)h);
    After:  struct acpi_madt *m = (struct acpi_madt *)h;
