fry358-b — Shell interactive typing fixes
Timestamp: 2026-02-27 18:20:26

Problem confirmed by user: window showed literal VT escape sequences and
typing produced no visible feedback:
  "Starting app...  TaterTOS64v3 shell
   [s [999:1HTaterTOS64v3 0:1:15 procs:17 [K [utater> "

Bug 1 — draw_taskbar() VT sequences rendered literally (shell.c)
  shell.c called draw_taskbar() each loop iteration before the prompt.
  draw_taskbar() emits \x1b[s, \x1b[999;1H, \x1b[K, \x1b[u (VT100).
  gfx_draw_text() has no ANSI interpreter — shows raw bytes as text.
  Fix: removed draw_taskbar() call from main loop. The GUI has its own
  taskbar; shell's VT-based one is meaningless in a TaterWin window.

Bug 2 — No local echo in gets_bounded (libc.c)
  gets_bounded() received chars from fry_read(0,...) but never wrote them
  back to stdout. User saw nothing while typing.
  Fix: after storing each printable char, fry_write(1, &c, 1) echoes it.
  Added backspace: \b or DEL (127) → back-erase with "\b \b".
  On \n: write "\n" before breaking so cursor advances before the prompt.

Bug 3 — 8-byte gate in process_window_output (gui.c)
  Outer while loop only ran when msgbuf >= 8 bytes (sizeof TW header).
  With echo active, each typed char is 1 byte. Chars accumulated silently
  until 8 had piled up, then rendered all at once — 8-char display lag.
  Fix: after process_window_output(), if 0 < msglen < 8, flush directly to
  textbuf. TW messages always arrive complete (< 128-byte poll chunk and
  written atomically), so sub-8-byte leftovers are always plain text.

Files changed:
  src/user/shell/shell.c   — removed draw_taskbar() call in main loop
  src/user/libc/libc.c     — gets_bounded: add echo + backspace handling
  src/user/gui/gui.c       — poll loop: flush sub-8-byte msgbuf remnants
