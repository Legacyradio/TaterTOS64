fry394 — Harden LPC IOE enable path for multi-hardware compatibility
Timestamp: 2026-02-28 20:34:20 UTC (2026-02-28 14:34:20 CST)

PROBLEM:
  fry393 enabled 00:1f.0 IOE bit 13 directly. That can be too broad for
  multi-hardware support if 00:1f.0 is not an Intel LPC/ISA bridge.

CHANGES:
1) src/kernel/acpi/extended.c
   - In ec_enable_lpc_decode(), added strict gating before any IOE write:
     - Read ID at 00:1f.0 offset 0x00 and require present device.
     - Require vendor 0x8086 (Intel).
     - Read class register at offset 0x08 and require class/subclass 0x06/0x01
       (bridge / ISA-LPC compatible bridge class).
   - If checks fail, skip write and leave diag sentinels at 0xFFFF.
   - Added kprint reason lines for absent/unsupported device.

WHY:
  This removes platform-specific assumptions and avoids touching undefined
  config semantics on non-Intel or non-LPC bridge hardware. It satisfies
  the no-shortcuts rule for compatibility hardening.
