fry407 - Post-rollback fix: runtime DMI PID probing for 8th-gen Dell + web-sourced guidance
Timestamp: 2026-03-01 01:53:05 UTC

CONTEXT:
  After full rollback of nuclear SBREG sweep (fry406), user confirmed target is
  Dell Precision on 8th-gen Intel i5. Web research showed DMI PID is not
  universal across Intel PCH families; hardcoding one PID can break PCR access.

CHANGES:
1) src/kernel/acpi/extended.c
   - Replaced single hardcoded PID with candidate set:
       PID_DMI_DEFAULT     = 0xEF
       PID_DMI_CANNONLAKE  = 0x88
       PID_DMI_PANTHERLAKE = 0x2F
   - Added pcr_get_dmi_pid_candidates():
       * Prioritizes 0x88 first for LPC device 0xA30E (8th-gen Dell class).
       * Otherwise keeps 0xEF first, then 0x88/0x2F.
   - Updated pcr_read_dmi_lpc_regs() to accept runtime pid argument.
   - Updated fallback SBREG BAR probing to test each bar with each PID candidate.
     Selected PID is persisted in g_pcr_pid_used for diagnostics.
   - Updated ec_mirror_pcr_ioe() to probe PID candidates at the chosen SBREG BAR
     and use the first plausible PID before writing IOE/IOD mirrors.
   - Nuclear sweep remains removed (from fry406).

WHY:
  On 8th-gen Intel Dell platforms, using PID 0xEF unconditionally can target the
  wrong sideband port. Runtime PID probing is safer and aligns with cross-gen
  Intel behavior differences observed in Linux/coreboot references.
