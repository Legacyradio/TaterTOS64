WHAT WAS CHANGED (fry420)
=========================================================================
After 29 consecutive iterations (fry391-fry419) that all failed to fix
the Dell EC, with sup=0 from fry419 proving that _REG's AML code does
NOT do EC I/O — this build simply SKIPS the _REG(3,1) call entirely.

The EC is confirmed alive (preREG=0x00) every boot.  It dies only after
_REG runs.  So: don't run _REG.

Changes:
  1) extended.c: Entire _REG block replaced with skip block
     - No _INI, no _REG bytecode dump, no suppression, no ECAV check
     - GPE enable kept (harmless)
     - Diagnostic kprints for EC status without _REG
     - Pre-probe delay reduced from ~200ms to ~50ms
     - Probe messages say "probe (no _REG)" instead of "recovery"
  2) sysinfo.c: Shows "REG=SKIP" instead of "REG=0"

WHAT TO LOOK FOR ON DELL BOOT
=========================================================================
Sysinfo panel:
  - "REG=SKIP" confirms this build
  - "EC: OK" = IT WORKED (EC survived without _REG)
  - "EC: FAIL" = EC still dies (something else is killing it)
  - "rcv=1" = normal probe succeeded
  - "rcv=0" = all probes failed

Three outcomes:
  1) EC: OK, rcv=1, Battery shows values → _REG was the killer. Done.
  2) EC: FAIL, sts=0xff → something before _REG kills it. Suspect
     ec_try_acpi_enable() in ec_setup_ports(). Next: remove that too.
  3) EC alive (sts!=0xff) but probe fails at OBF → EC needs _REG to
     accept READ commands but _REG kills it. Need to fix AML interpreter.
