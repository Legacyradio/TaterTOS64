REGISTER INFORMATION (from Intel EDK2 CoffeelakeSiliconPkg/PchRegsLpc.h):

  PCI Config (D31:F0):
    0xDC = R_ESPI_CFG_PCBC (Peripheral Channel BIOS Control)
      Bit 2 = ESPI_EN (eSPI Enable pin strap): 1=eSPI mode, 0=LPC mode

  PCR Sideband (eSPI PID space):
    0x4000 = SLV_CFG_REG_CTL (indirect slave register access)
      Bit 31 = SCRE (access enable)
      Bits 30:28 = SCRS (status: 7 = no error/complete)
      Bits 17:16 = SCRT (0=read, 1=write)
      Bits 11:0 = SCRA (slave register address)
    0x4004 = SLV_CFG_REG_DATA (slave register readback)
    0x4020 = PCERR_SLV0 (Peripheral Channel Error, bit 24 = unsupported req)
    0x4030 = VWERR_SLV0 (Virtual Wire Error)
    0x4040 = FCERR_SLV0 (Flash Channel Error)
    0x4050 = LNKERR_SLV0 (Link Error)
      Bit 31 = SLCRR (link/slave channel recovery active)
      Bit 20 = LFET1S (fatal error type 1 status)
    0xC00C = CFG_VAL (bit 0 = eSPI enabled strap, non-FF = valid PID)

  eSPI Slave Registers (accessed indirectly via SLV_CFG_CTL/DATA):
    0x0008 = General/Device Configuration
      Bits [2:0] = Operating Frequency
        000=20MHz, 001=25MHz, 010=33MHz(Gen3), 011=50MHz, 100=66MHz(Gen4)
      Bits [4:3] = I/O Mode Support
        00=Single, 01=Single+Dual, 10=Single+Quad, 11=Single+Dual+Quad
    0x0010 = Peripheral Channel Cap & Config
    0x0020 = Virtual Wire Cap & Config

  PID for eSPI on Coffee Lake-H is UNKNOWN (not Cannon Lake 0x72).
  Code tries three candidates: 0x72, 0xC7 (PID_LPC), 0x6E.
  The correct PID returns non-FF at offset 0xC00C.

WHAT THIS BUILD DOES:

  1) READ ESPI_EN STRAP (PCI config, no PCR needed)
     Reads D31:F0 offset 0xDC (PCBC). Bit 2 = ESPI_EN.
     If 1: hardware is in eSPI mode (EC uses eSPI, not LPC).
     If 0: hardware is in LPC mode (eSPI not relevant).
     This is the FIRST thing to check.

  2) SCAN FOR CORRECT eSPI PID
     Reads PCR[PID]+0xC00C (CFG_VAL) at three PID candidates.
     The correct PID returns a non-0xFFFFFFFF, non-zero value.
     Selects the first PID that passes this test.

  3) READ eSPI ERROR REGISTERS
     At the found PID, reads:
       PCERR_SLV0 (0x4020): peripheral channel errors
       VWERR_SLV0 (0x4030): virtual wire errors
       FCERR_SLV0 (0x4040): flash channel errors
       LNKERR_SLV0 (0x4050): link errors (fatal, recovery)
     If any error bits are set, the eSPI bus has problems that would
     explain the floating bus on EC ports.

  4) READ SLAVE GENERAL CONFIG (indirect access)
     Issues a read command via SLV_CFG_CTL (0x4000) for slave register
     0x0008 (General/Device Configuration). Reads result from SLV_CFG_DATA
     (0x4004). Decodes:
       - Operating Frequency: 20/25/33(G3)/50/66(G4) MHz
       - I/O Mode: Single / Dual / Quad
     This tells us EXACTLY what Gen and I/O mode the BIOS negotiated.

  5) ALSO: LPC GEN3_DEC/GEN4_DEC + PCR IOE FIX (from rev1)
     - Reads LPC PCI config 0x8C and 0x90 (Generic I/O Decode Range 3 & 4)
     - Fixed PCR IOE read bug: was showing 0xFFFFFFFF when BIOS already
       had IOE set (code skipped the read, not just the write)

  6) SYSINFO DISPLAY (WIN_H increased to 560)
     Y=320: LPC8C and LPC90 values
     Y=400: eSPI line 1: en=N pid=NN freq io=XX gc=NNNNNNNN
       en=1 + cyan: eSPI mode confirmed
       en=0 + red: LPC mode (eSPI not relevant)
     Y=416: eSPI line 2: err: pc=NN vw=NN ln=NN
       Gray if all zero (no errors), orange if any set (bus errors)

EXPECTED ON DELL BOOT:

  STEP 1: Check ESPI_EN strap
    en=1 → Dell is using eSPI mode (expected for Coffee Lake-H)
    en=0 → Dell is using LPC mode (unexpected, changes diagnosis)

  STEP 2: PID scan result
    If PID=0x72 returns valid CFG_VAL: Cannon Lake PID works on this PCH
    If PID=0xC7 returns valid CFG_VAL: eSPI shares PID with LPC controller
    If PID=0x6E returns valid CFG_VAL: alternate PID
    If ALL PIDs return 0xFFFFFFFF: eSPI PCR inaccessible (need more PIDs)

  STEP 3: Error registers
    PCERR=0 VWERR=0 LNKERR=0: no bus errors → bus is clean
    Any non-zero: eSPI bus has errors → THIS is why EC doesn't respond
    LNKERR bit 31 (SLCRR): link recovery → bus keeps resetting
    LNKERR bit 20 (LFET1S): fatal error → bus is dead

  STEP 4: Slave General Config (THE KEY DIAGNOSTIC)
    frequency field:
      010 = 33MHz (Gen 3) — standard for many platforms
      100 = 66MHz (Gen 4) — faster, stricter signal requirements
    io_mode field:
      00 = Single I/O — basic, most compatible
      01 = Single+Dual — 2x bandwidth
      10 = Single+Quad — 4x bandwidth
    If read succeeds: BIOS negotiated config is visible
    If read times out: slave isn't responding to in-band config reads either
      → eSPI bus is completely non-functional

  SERIAL OUTPUT to look for:
    "ACPI: eSPI PCBC(0xDC)=0xNN ESPI_EN=N"
    "ACPI: eSPI PID=0xNN CFG_VAL(0xC00C)=0xNN" (one per candidate)
    "ACPI: eSPI using PID=0xNN CFG_VAL=0xNN"
    "ACPI: eSPI PCERR=... VWERR=... FCERR=..."
    "ACPI: eSPI LNKERR=... SLV_CFG_CTL=..."
    "ACPI: eSPI slave GenCfg=0xNN freq=NNMHz io=XX SCRS=N"
    OR: "ACPI: eSPI slave GenCfg read timeout"
    AND possibly: "ACPI: eSPI LINK RECOVERY ACTIVE" or "FATAL ERROR"
