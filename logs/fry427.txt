WHAT CHANGED (why)
================================================================
- In `src/kernel/acpi/extended.c`, added `invoke_path_method()` to call ACPI methods while treating no-return methods as successful invocations.
- Switched EC handshake calls from return-object-based checks to method-invocation checks:
  - `_REG(3,1)` now logs success even when it returns no AML object (void method).
  - `_INI` now does the same.
- Added explicit log variants for returned vs void methods so boot logs distinguish both paths.
- Corrected `_GPE` fallback diagnostics: when `_GPE` is missing/unreadable and fallback `111` is used, `g_ec_gpe_found` now remains `0` instead of being incorrectly set to `1`.

RATIONALE
================================================================
- Prior logic marked `_REG` and `_INI` as failed if `aml_eval_with_args()` returned `NULL`.
- In ACPI, control methods often return no object; `NULL` can still mean the method executed.
- This produced false "_REG invocation failed" output in Dell boot logs and obscured real EC/eSPI issues.
