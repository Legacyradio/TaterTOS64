Code changes:

1) src/kernel/acpi/extended.c
   A) Added eSPI PID selection helpers:
      - lpc_is_intel_300_series()
      - espi_get_pid_candidates()
      - espi_select_pid()
      WHY:
        Prior logic accepted the first non-FF CFG_VAL candidate, which can
        select a suboptimal PID on some systems and produce misleading
        diagnostics/recovery behavior.

   B) Hardened PID selection strategy:
      - Uses candidate scoring with strap match and platform bias.
      - Prefers PID 0xC7 on Intel 300-series-class LPC/eSPI device IDs.
      - Logs candidate cfg/score decisions for visibility.
      WHY:
        Improves deterministic PID choice on Coffee Lake-era systems while
        preserving runtime detection for other platforms.

   C) Upgraded espi_clear_errors() from single-pass to multi-pass:
      - ESPI_CLEAR_MAX_PASSES = 3
      - ESPI_CLEAR_SETTLE_LOOPS = 10000000 (~10ms target settle)
      - Re-read and log register state after each pass and final state.
      WHY:
        Sticky W1C states can persist briefly; repeated clear+settle improves
        chance of full recovery before EC probe retry.

   D) Recovery path after espi_clear_errors(1):
      - Replaced very short settle loop with the same ~10ms settle helper.
      WHY:
        Aligns post-clear timing with observed need for controller/link settle.

2) src/user/shell/shell.c
   - acpibreak output label changed from "pid" to "pcrpid".
   WHY:
     Avoids confusion between eSPI PCR PID and Virtual Wire index.

3) src/user/apps/sysinfo.c
   - sysinfo eSPI label changed from "pid" to "pcrpid".
   WHY:
     Same diagnostic clarity improvement as shell output.
