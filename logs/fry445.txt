## Key Diagnostic Analysis

### THE SMOKING GUN: `rdy=0`
- PC_CAP=0x00001117, bit 16 (ready) = 0 -> Peripheral Channel NOT READY
- VW_CAP=0x00001117, bit 16 (ready) = 0 -> Virtual Wire Channel NOT READY
- This means the EC slave has NOT asserted channel readiness on eSPI
- Without PC ready, I/O cycles to 0x62/0x66 are NOT tunneled -> floating bus -> 0xFF

### eSPI Error Register Analysis:
- PCERR before clear: 0x00001200, status bits: 0x00001000 (bit 12 = NFES non-fatal error)
- After W1C clear: 0x00000200 (errors cleared, bit 9 = some enable/default bit)
- VWERR: 0x00000000 (clean)
- FCERR: 0x00040080 (flash channel errors present but not status bits)
- LNKERR: 0x0060FF00 (link errors present but not status bits)
- All STATUS bits read 0x00000000 after clear -> errors cleared successfully

### What's Working:
- eSPI bus IS enabled (espi en=1)
- PCR PID 0x72 IS correct for Cannon Lake/Coffee Lake PCH
- eSPI error clearing IS working (W1C succeeds)
- Slave register reads ARE working (rdok=0x07 = genCfg+pcCap+vwCap all read OK)
- LPC IOE bits ARE set (0x3f0f includes bits 10+11 for 0x60/64 and 0x62/66)

### What's NOT Working:
- EC slave has rdy=0 on both PC and VW channels
- EC ports 0x62/0x66 and 0x68/0x6C all read 0xFF (floating bus)
- Battery _STA returns 0x00000000 (not present/functional - likely because EC is dead)
- Backlight _STA returns 0xFFFFFFFF (probably hardcoded or doesn't need EC)

## Critical Finding: Zephyr RTOS eSPI Driver (Microchip XEC)

The Zephyr project has an open-source eSPI slave driver for Microchip XEC
(same silicon family as MEC152x). Source: espi_mchp_xec_v2.c

### Channel Ready Assertion Sequence (from EC firmware side):
1. Host de-asserts ESPI_nRESET -> espi_rst_isr() fires
2. OOB/Flash channels re-initialized, bus interrupts re-enabled
3. Host de-asserts PLT_RST (Platform Reset) via Virtual Wire
4. vw_pltrst_handler() fires -> calls setup_espi_io_config()
   - Configures I/O BAR: regs->IOHBAR[IOB_IOC] = (host_addr << 16) | HOST_VALID
   - This maps host addresses like 0x62/0x66 to EC internal registers
5. Host sends SET_CONFIGURATION to enable Peripheral Channel
6. espi_pc_isr() detects MCHP_ESPI_PC_STS_EN_CHG (channel enable change)
7. espi_pc_isr() calls setup_espi_io_config() again
8. espi_pc_isr() writes regs->PCRDY = 1  *** THIS ASSERTS READY ***
9. Similar flow for VW: espi_vw_chan_en_isr() sets regs->VWRDY = 1

### Key Insight: The EC firmware must explicitly write PCRDY=1
- The ready bit is NOT automatic hardware
- It's asserted by EC firmware in response to host channel enable
- If EC firmware never runs this code path, rdy stays 0 forever
- This explains why we see rdy=0: the EC firmware hasn't completed
  its channel initialization sequence

### Possible Root Causes for rdy=0:
A. EC firmware is in a pre-boot state waiting for something from UEFI
   that we never sent (because we took over from UEFI)
B. PLT_RST virtual wire was never properly de-asserted by our OS
C. The host (PCH) never sent SET_CONFIGURATION to enable the PC channel
D. EC firmware crashed or is in a reset loop
E. Dell's EC firmware has proprietary initialization beyond standard eSPI

## Dell-Specific Issues Found

### BIOS Version Problems:
- BIOS 1.12.1 caused ACPI errors on Linux (Dell community reports)
- Users had to revert to BIOS 1.11 to fix ACPI/EC issues
- Dell BIOS updates include "Embedded Controller Engine Firmware" updates
- EC firmware is tightly coupled with BIOS version

### Dell EC Firmware Encryption:
- Dell ships laptops with ECs that have ENCRYPTED firmware
- Decryption keys are FUSED inside the EC chip
- Cannot replace EC chip without matching keys (right-to-repair issue)
- This means we cannot reflash or modify EC firmware

### Dell Community Reports (Precision 7530):
- Multiple users report ACPI BIOS errors on Linux
- Issues traced to BIOS firmware, not Linux kernel
- EC firmware updates bundled with BIOS updates
- "EC Firmware Update Programming Tool" exists (driver ID: wnxxt)

## Summary

The root cause is almost certainly that TaterTOS64v3 is not completing the
eSPI channel initialization handshake that the EC firmware expects. The
Zephyr driver analysis shows the EC waits for:
  1. ESPI_nRESET de-assertion
  2. PLT_RST de-assertion (Virtual Wire)
  3. Host SET_CONFIGURATION enabling Peripheral Channel
  4. Only THEN does EC firmware write PCRDY=1

We are likely missing step 2 or 3 (or both). UEFI firmware normally handles
this during POST, but when we take over, the EC may have been reset or may
need these signals re-sent. The most promising approach is to send the
PLT_RST VW de-assertion and then SET_CONFIGURATION for PC channel enable
via the PCH's eSPI master registers.
