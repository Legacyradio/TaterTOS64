- User requested making app launch "play nice" on Dell without removing NVMe support.
- Goal: keep GUI/shell launch responsive even when NVMe secondary probing is slow/flaky.

CHANGES APPLIED
---------------
1) src/kernel/main.c
   - Added explicit minimal fallback-first launch order:
     - Try `/GUI.FRY` first.
     - If that fails, try `/SHELL.TOT`.
     - Only then attempt broader path variants.
   - This ensures first-launch usability is not blocked by broader path scanning.

2) src/kernel/fs/vfs.c
   - Tightened `vfs_try_recover_mount(...)` trigger rules.
   - Secondary recovery now runs only for exact mountpoint paths (`/nvme`, `/nvmeN`),
     not arbitrary `/nvme/...` file opens.
   - Prevents app file opens from implicitly triggering storage recovery scans.

3) src/user/gui/gui.c
   - Start-menu discovery is now primary-first:
     - Seed minimal app list immediately (`SHELL`, `SYSINFO`, `UPTIME`, `PS`, `FILEMAN`).
     - Scan primary paths (`/`, `/fry`, `/FRY`, `EFI/...`) first.
     - Scan `/nvme*` only if secondary mounts are already present.
   - Spawn path resolution is now primary-first with secondary fallback only when mounts exist.
   - Added one-shot launch watchdog (`LAUNCH_WATCHDOG_MS=5000`):
     - If a spawned app shows no output/window progress by deadline, GUI marks it done,
       kills the stuck process, and stays responsive.
