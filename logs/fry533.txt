TaterTOS64v3 - fry533 — Fix app usability regression from post-sti NVMe mount
========================================================================

TIMESTAMP
---------
2026-03-05 04:07:22 UTC

CONTEXT
-------
- fry532 introduced automatic secondary NVMe mount in live mode by running
  `vfs_mount_secondary()` after `sti` in a kernel worker.
- Reported regression: app launches became unusable/stalled after fry532.

ROOT CAUSE
----------
- VFS mount-list operations are currently unsynchronized.
- Performing secondary mount after `sti` allowed concurrent runtime VFS access
  (app launch file opens/stats) while mount-list mutation was in flight.
- That race can break or stall app launch behavior.

FIX APPLIED
-----------
1) `src/kernel/main.c`
   - Keep live-mode auto secondary mount behavior.
   - Move deferred `/nvme` mount execution to pre-`sti` (late init, still
     single-threaded from a VFS perspective).
   - Remove post-`sti` nvme mount worker scheduling path.

EXPECTED IMPACT
---------------
- Apps regain stable launch behavior (no runtime VFS mount mutation race).
- `/nvme` is still auto-mounted in live mode, but before scheduler start.
- Tradeoff: slight boot-time delay possible before first user task runs.
