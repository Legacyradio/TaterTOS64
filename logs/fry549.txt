CONTEXT
-------
- Follow-up review flagged a potential leak in `vmd.c` from:
  `pci_get_devices(&count)` with no corresponding free.

WHY
---
- `pci_get_devices()` returns PCI-core static storage (`pci_info[256]`) in
  `src/drivers/pci/pci_core.c`, not heap memory.
- There is no allocation to release, but API/call-site types were mutable and
  could imply ownership to readers.

CHANGE APPLIED
--------------
- Updated API and call sites to make ownership explicit and non-mutable:
  - `src/drivers/pci/pci.h`
    - `pci_get_devices()` now returns
      `const struct pci_device_info *`
    - Added comment: caller must not free/modify returned storage.
  - `src/drivers/pci/pci_core.c`
    - Updated function definition return type to `const`.
  - `src/drivers/storage/vmd.c`
    - Updated both local `devs` pointers to `const`.
  - `src/kernel/drivers/platform.c`
    - Updated local `devs` pointer to `const`.
  - `src/kernel/acpi/extended.c`
    - Updated both local `devs` pointers to `const`.

EFFECT
------
- Confirms no memory leak exists for `pci_get_devices()` usage in VMD probe.
- Prevents accidental mutation/free assumptions at compile-time.
