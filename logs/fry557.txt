WHY
---
- Mid-transfer failures need bounded retry + recovery to avoid hard fail on
  transient queue/interrupt issues.
- Improves robustness when IO queue state or completion mode gets unhealthy.

CHANGE APPLIED
--------------
- File updated: `src/drivers/storage/nvme.c`
  - Added constants:

    - `NVME_IO_SUBMIT_RETRIES` (set to 3)
  - Added `nvme_recover_io_queue()` helper:
    - Falls back from IRQ completions to polling mode when recovery starts.
    - Issues best-effort Admin commands to delete IO SQ/CQ for queue id 1.
    - Recreates IO queue via existing `nvme_create_io_queues()` path.
  - Updated `nvme_io_rw()` submit path:
    - Added bounded retry loop for `submit_cmd(&io_q, ...)`.
    - Emits retry-aware trace event (`submit_fail`, attempt count).
    - Calls `nvme_recover_io_queue()` between attempts.
    - Returns failure only after retries/recovery attempts are exhausted.

EFFECT
------
- Adds concrete retry and queue-recovery behavior for failed IO submits.
- Avoids immediate terminal failure on first transient submit timeout/failure.
- Preserves existing command chunking and PRP handling flow.
