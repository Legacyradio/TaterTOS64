CONTEXT
-------
- Review flagged potential leak behavior from `aml_eval(hid_path)` in ACPI
  device enumeration.

WHY
---
- AML object allocator is monotonic (`aml_obj_arena` in `aml_types.c`) with no
  per-object free API.
- Best mitigation here is to avoid calling `aml_eval()` when `_HID` object is
  already present in namespace node storage.

CHANGES APPLIED
---------------
- File updated:
  - `src/drivers/acpi_bus/acpi_bus.c`
    - In `create_acpi_device()`, lookup `_HID` node with
      `ns_find_child(node, {'_','H','I','D'})`.
    - If `_HID` node already has `node->object`, use it directly.
    - Only build path and call `aml_eval()` as fallback when object is absent.

EFFECT
------
- Eliminates unnecessary AML evaluations/allocations for common static `_HID`
  declarations (NameOp-backed objects already in namespace).
- Keeps compatibility for `_HID` cases that require runtime evaluation.
