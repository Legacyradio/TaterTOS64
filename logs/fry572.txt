CONTEXT
-------
- Review flagged possible truncation risk for NVMe PRP addresses (`prp1/prp2`)
  if DMA buffers are above 4 GiB.

WHY
---
- Audit showed runtime path already uses 64-bit PRP fields and 64-bit physical
  address values, but this should be locked down so future edits cannot
  accidentally regress to 32-bit PRP packing or assignment.

CHANGES APPLIED
---------------
- File updated:
  - `src/drivers/storage/nvme.c`
    - Added compile-time command-layout guards:
      - `prp1` width is 64-bit
      - `prp2` width is 64-bit
      - `prp1` offset is 24
      - `prp2` offset is 32
      - command size is 64 bytes
    - Added helper `nvme_cmd_set_prps(struct nvme_cmd *cmd, uint64_t prp1,
      uint64_t prp2)` and switched PRP assignments to this helper in:
      - Identify command build
      - Create IO CQ/SQ command build
      - I/O read/write command build

EFFECT
------
- PRP address fields are now explicitly guarded against type/layout regressions
  at compile time.
- All command PRP writes in the touched paths use explicit `uint64_t`
  assignments (no 32-bit cast path).
