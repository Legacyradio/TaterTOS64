CONTEXT
-------
- Review flagged brittle early physmap coverage risk if RAM/ACPI/boot data
  lives above low fixed limits.
- Existing logic hard-capped physmap bootstrap to `early_identity_limit`,
  which can miss high-memory regions on modern UEFI systems.

WHY
---
- Early kernel access to handoff-backed structures must not assume a fixed
  1 GiB/4 GiB window.
- Physmap bootstrap should reflect actual UEFI memory descriptors so early
  consumers (filesystem/ACPI paths) can safely touch mapped physical ranges.

CHANGES APPLIED
---------------
- File updated:
  - `src/kernel/mm/vmm.c`
    - Added `PAGE_SIZE_2M` constant and used it in `map_range_2m()`.
    - Added `is_physmap_bootstrap_type()` helper to select descriptor types
      safe/useful for bootstrap mapping.
    - Reworked `vmm_init()` physmap bootstrap:
      - Iterates EFI memory descriptors.
      - Validates page count overflow before byte conversion.
      - Aligns each selected descriptor to 2 MiB boundaries.
      - Maps each selected range directly into the physmap window.
      - Tracks `mapped_max` from mapped ranges instead of clamping to
        `early_identity_limit`.
    - Added fallback: if descriptor-driven mapping produces no range, map
      `[0, early_identity_limit)` as before.
    - Set `physmap_max = mapped_max`.
    - Increased default `early_identity_limit` from 1 GiB to 4 GiB.

EFFECT
------
- Physmap bootstrap no longer depends on a fixed low-memory cap when a valid
  UEFI memory map is present.
- High physical ranges described by UEFI are now covered during early bring-up,
  reducing early page-fault risk when touching high-memory tables/data.
