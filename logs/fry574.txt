CONTEXT
-------
- Review requested three checks:
  - Ensure `kmalloc` is not fed a truncated 32-bit length when source size is
    naturally wider.
  - Verify variable-depth reverse loops are using an unsigned/address-sized
    pattern on 64-bit builds.
  - Ensure NVMe doorbell stride math derived from `CAP.DSTRD` stays 64-bit.

WHY
---
- NTFS should not narrow a sector-sized allocation argument before calling a
  64-bit `kmalloc`.
- NVMe register-address math should not depend on 32-bit temporaries when
  deriving doorbell offsets from stride and queue id.
- Reverse loop audit needed to confirm the earlier `depth` underflow issue was
  actually resolved rather than still present elsewhere.

CHANGES APPLIED
---------------
- Files updated:
  - `src/kernel/fs/ntfs.c`
    - Removed two explicit `(uint32_t)` casts on `dev_sec_size` before
      `kmalloc()` in `ntfs_dev_read_bytes()`.
  - `src/drivers/storage/nvme.c`
    - Widened `db_stride` in `struct nvme_device` to `uint64_t`.
    - Widened `mmio_read32()`, `mmio_write32()`, and `mmio_read64()` offset
      parameters to `uint64_t`.
    - Reworked `doorbell_sq()` / `doorbell_cq()` to compute offsets in
      `uint64_t`.
    - Widened `g_db_stride = 4ULL << dstrd`.

VERIFICATION NOTES
------------------
- Reverse-loop audit:
  - `src/drivers/acpi_bus/acpi_bus.c` already uses
    `for (size_t i = (size_t)depth; i-- > 0;)`.
  - No remaining `for (int i = depth - 1; ...)` pattern was found in `src/`.
  - Remaining reverse `int` loops found during audit are fixed small-count GUI
    or formatting loops, not variable-depth/address-sized traversals.

EFFECT
------
- NTFS no longer explicitly truncates the sector allocation size before
  passing it to `kmalloc`.
- NVMe doorbell register offset math now stays 64-bit from `CAP.DSTRD`
  decoding through MMIO access helpers.
- The earlier `depth` reverse-loop issue remains fixed; no extra patch was
  required for that audit item.
