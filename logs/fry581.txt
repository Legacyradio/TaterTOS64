CONTEXT
-------
- Bare-metal boot is now stable, and shutdown/reboot work on the Dell Precision
  7530.
- Remaining user-visible issue: File Manager showed no NVMe storage on real
  hardware when booted from live media.
- Follow-up request: make the storage/mount state queryable from the shell with
  a clear command, not just hidden debug behavior.

WHY
---
- Live boot kept `/` on ramdisk and explicitly skipped NVMe secondary
  auto-mount, so File Manager had no `/nvme*` mountpoints to display even when
  NVMe detection was working.
- Storage state already existed behind syscalls, but the shell only exposed it
  through the non-obvious `vfsdbg`/`nvmefix` commands.

CHANGES APPLIED
---------------
- Files updated:
  - `src/kernel/main.c`
    - Replaced the live-mode "skip NVMe secondary auto-mount" path with
      `vfs_mount_secondary(nvme_bd, "/nvme")`.
    - Added explicit boot log outcomes:
      - `init: live mode, NVMe secondary auto-mount ready`
      - `init: live mode, NVMe secondary auto-mount unavailable`
  - `src/user/shell/shell.c`
    - Split storage and mount reporting into dedicated helpers.
    - Added user-facing commands:
      - `storage`
      - `mounts`
      - `mountdbg`
    - Updated `help` output to list the new commands.
    - Kept existing `vfsdbg [path]` and `nvmefix [path]` flows intact.

EXPECTED BEHAVIOR
-----------------
- On live boot, supported secondary NVMe partitions (FAT32 / NTFS / ToTFS) are
  now mounted under `/nvme`, `/nvme1`, ... while the root filesystem remains on
  ramdisk.
- File Manager can discover those mountpoints via `fry_mounts_info()`.
- Shell diagnostics are now directly accessible:
  - `storage`
  - `mounts`
  - `mountdbg`
  - `vfsdbg`
  - `nvmefix`
