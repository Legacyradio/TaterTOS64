CONTEXT
-------
- User reported the remaining issue is on bare metal; QEMU logs are not relevant
  for this step per plan rule 11.
- Current boot diagnostics already bracket the failure around the first scheduler
  handoff:
  - stage 32 / `K_BOOT_USER_ENQUEUED`
  - stage 33 / `K_FIRST_TICK`
  - stage 34 / `K_FIRST_SWITCH`
  - stage 35 / `K_FIRST_USER`
- Review of the live tree found one real lifecycle bug unrelated to the earlier
  Gemini theory:
  - `acpi_events_init()` was creating and enqueuing a kernel worker before
    `process_init()` and `sched_init()` later reinitialized the process and
    scheduler state.
- Bare-metal startup was also still depending on the first LAPIC timer tick to
  perform the initial handoff into userspace.

WHY
---
- Creating scheduler-managed threads before the process layer is initialized is
  invalid lifecycle ordering and can corrupt or discard queued worker state.
- For the first boot userspace jump, relying on the first external timer tick
  adds avoidable hardware timing sensitivity on the Dell 7530.
- The goal of this step is to make the first userspace handoff deterministic on
  bare metal without introducing Dell-specific behavior.

CHANGES APPLIED
---------------
- Files updated:
  - `src/kernel/acpi/events.h`
    - Added `acpi_events_start_worker()` declaration.
  - `src/kernel/acpi/events.c`
    - Moved GPE worker creation out of `acpi_events_init()`.
    - Added `acpi_events_start_worker()` so the worker is only created after the
      process/scheduler subsystem is live.
  - `src/kernel/main.c`
    - Added a direct `sched_tick()` immediately after boot userspace is queued
      and before `sti`.
    - Deferred `acpi_events_start_worker()` until after that first direct
      scheduler handoff.

EXPECTED EFFECT
---------------
- The first boot user process no longer depends on the first LAPIC timer tick to
  receive its initial scheduler handoff.
- Early ACPI GPE worker creation no longer races against later
  `process_init()`/`sched_init()` reinitialization.
- Existing post-launch markers remain valid for deeper downstream diagnosis:
  - 32 = boot user enqueued
  - 33 = first BSP timer tick
  - 34 = first context switch
  - 35 = first scheduled user process
  - 36 = first user syscall
  - 37 = first `INIT.FRY` syscall
  - 38 = init spawned GUI
  - 39 = GUI reached framebuffer syscall path
- Marker interpretation changes slightly for this artifact:
  - stages 34 and 35 may now occur before stage 33, because the first handoff
    can happen via the direct pre-`sti` `sched_tick()`.
