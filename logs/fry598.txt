CONTEXT
-------
- Hardware report: boot reaches 37 white squares, then shows many red spawn
  squares (about 40 observed).
- Per fry597, red row 1 squares mean INIT.FRY is alive and SYS_SPAWN keeps
  failing while it retries GUI/SHELL launch paths.
- INIT loops forever through 7 GUI paths and 7 shell paths, so many red
  squares mean repeated launch failures, not a one-shot halt.

DIAGNOSIS
---------
- EFI loader builds the live ramdisk from discovered userspace payload files.
- Before this change, alternate EFI filesystem handles were scanned only if
  pass 1 found zero files on the primary handle.
- On firmware that exposes a partial primary handle, the loader can find
  INIT.FRY there but miss GUI.FRY and SHELL.TOT even though those files exist
  on another EFI filesystem handle.
- That failure mode matches the Dell symptom exactly: INIT boots, then every
  child spawn fails forever.

FIX
---
- Updated src/boot/efi_loader.c so pass 2 scans alternate EFI Simple File
  System handles whenever fry_found is less than the full expected payload
  count, not only when fry_found is zero.
- Why: a partial primary filesystem result must not block discovery of the
  remaining live userspace payloads needed by INIT.

FILES CHANGED
-------------
- src/boot/efi_loader.c
  - Changed alternate SFS scan gate from "fry_found == 0" to
    "fry_found < FRY_FILE_COUNT".
