- Meaning:
  - the canonical live-session payload paths are present in the mounted root
  - the root-level fallback aliases are absent from the packed ramdisk layout
- Since INIT launches through `/system/...` and `/apps/...` first, this report
  moves the investigation away from "GUI/SHELL missing from live root" and
  toward the remaining `ELF_LOAD_ERR_OPEN` path itself.

ROOT CAUSE TARGETED
-------------------
- `vfs_open()` allocated each `struct vfs_file` with `kmalloc()`.
- Kernel `kfree()` is still a no-op bump allocator free stub.
- Result:
  - every successful open/close permanently consumed heap,
  - and all later opens still depended on whatever heap headroom remained.
- That is a bad fit for the current boot path because userspace launch and file
  open churn happen after a large amount of ACPI / driver initialization has
  already consumed the tiny early heap.

FIX
---
- File changed:
  - `src/kernel/fs/vfs.c`
- Replaced heap-backed `struct vfs_file` allocation with a reusable static pool
  of 256 open-file objects guarded by the existing VFS lock.
- `vfs_open()` now:
  - allocates from the pool,
  - zeros the slot before use,
  - logs `VFS: open file pool exhausted path=...` if the pool is full.
- `vfs_close()` now:
  - returns the slot to the pool instead of calling no-op `kfree()`.

WHY
---
- This removes one concrete reason `vfs_stat()` could succeed while
  `vfs_open()` later fails in the live boot path.
- It also stops open/close traffic from silently burning the limited bump heap.
- The user-reported 4-green / 2-red row is still informative:
  canonical files are there; if Dell still shows spawn-open failures after
  this patch, the next suspect is no longer missing payload population.
