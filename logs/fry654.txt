  User provided bare-metal shell output from Dell Precision 7530:
    - `wd` showed `step=9 (alive) rc=-1`

    - `FH_LOAD_STATUS=0xffffffff`

    - `RFH_USED_WIDX=0x00000000`
  `ws` showed:
    - `wifi: driver=offline`
    - `FAIL @ alive (step 9) rc=-1`

WHY:
  The firmware upload path appears to complete (`FH_LOAD_STATUS=0xffffffff`) and
  an interrupt has fired (`irq_count=1`), but the RX completion path never
  surfaces a usable ALIVE packet to the driver.

  The concrete bug found in `src/drivers/net/wifi_9260.c` was that the 9000-series
  multi-queue RX free ring was populated with raw physical addresses only:
    `rxbd_ring[i].addr = rx_buf_phys[i]`

  For this hardware path, the free ring entry must carry:
    DMA address | non-zero 12-bit VID

  The used ring then reports that VID back on completion. The old code treated
  the used-ring low 12 bits as a direct RX buffer index, which is wrong for this
  encoding and can leave the host polling the wrong buffers forever.

CHANGES:
  File changed:
    - `src/drivers/net/wifi_9260.c`

  Code updates:
    1. Documented the RX BD format for 9000-series MQ RX.
    2. Added `iwl_rx_encode_bd_addr(phys, vid)`.
    3. Added `iwl_rx_used_entry_to_idx(used_entry)` to decode VID -> buffer index.
    4. Changed FRBDCB population to post `phys | (i + 1)` instead of raw `phys`.
    5. Updated all RX consumers to decode used-ring entries through the helper:
       - `iwl_wait_alive()`
       - `iwl_wait_resp()`
       - `iwl_rx_poll()`
    6. Added invalid-entry debug prints so future `wd`/runtime logs will show
       malformed used-ring completions explicitly.

RATIONALE:
  This fix matches the observed failure mode better than the older context-info
  experiments in the repo:
    - upload completed
    - RFH queue was armed
    - ALIVE still never became visible to host code
  If the device completes RX using VID-tagged entries while the host interprets
  them as plain ring indices, step 9 stalls exactly as seen.
