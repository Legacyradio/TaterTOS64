    - `wd` showed `step=4 (tlv-parse) rc=-5`
    - `ws` showed `wifi: driver=offline`
    - boot probes already indicated earlier init reached firmware load

WHY:
  The TLV parser in `src/drivers/net/iwlwifi_fw.c` still enforced a legacy
  requirement:
    - CPU1 instruction TLV (`IWL_UCODE_TLV_INST`) must be present

  That is wrong for the embedded 9260 firmware blob in this tree because the
  runtime image can be expressed only as secure multi-section TLVs:
    - `SEC_RT`
    - optional `SEC_INIT`
    - optional paging TLVs

  In that case, `tlv-parse` should succeed and let the runtime uploader consume
  `pieces->sec_rt`. Returning `-5` at step 4 prevents the already-implemented
  secure-section upload path from ever running.

CHANGES:
  File changed:
    - `src/drivers/net/iwlwifi_fw.c`

  Code updates:
    1. Kept the existing legacy `INST` validation for legacy-format firmware.
    2. Changed the final parse check so `-5` is returned only if BOTH are absent:
       - legacy CPU1 `INST`
       - secure runtime sections (`SEC_RT`)
    3. Added a debug line that explicitly reports when the parser is accepting
       a `SEC_RT` runtime image without a legacy `INST` TLV.

RATIONALE:
  This matches the rest of the driver as it exists now:
    - TLV parsing records all `SEC_RT` sections
    - later code detects secure multi-section firmware
    - runtime upload walks `pieces->sec_rt`

  Without this parser change, the driver rejects a valid secure-format firmware
  image before DMA, TFH, upload, or ALIVE handling can begin.
