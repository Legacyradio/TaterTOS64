SESSION CONTEXT:
  User provided Dell bare-metal output:
    - `wd`: step=9 (alive) rc=-1, irq_count=1
    - last_irq_csr=0x02000001 → BIT(0)=ALIVE + BIT(25)=SW_ERR
    - last_irq_fh=0x00000000 → no FH RX activity
    - rx_closed=0, rx_finished=0 → RFH never delivered anything
    - upload_fh_st=0xffffffff → FW upload completed
    - `ws`: wifi: driver=offline, FAIL @ alive (step 9) rc=-1

ROOT CAUSE:
  The firmware booted (CSR_INT_BIT_ALIVE fired) but immediately crashed
  (CSR_INT_BIT_SW_ERR). No FH RX interrupt or RX completion occurred,
  so the ALIVE notification was never delivered to the host.

  Comparison against Linux 7.0-rc2 iwlwifi revealed that Linux calls
  iwl_pcie_tx_init() BEFORE firmware upload, setting up:

    1. FH_KW_MEM_ADDR_REG (0x197C): keep-warm buffer DMA address >> 4
       - Keeps NIC's FH subsystem from entering low-power mode
    2. FH_MEM_CBBC_QUEUE(0) (0x19D0): TFD ring base for command queue >> 8
       - Gen1 uses FH_MEM_CBBC, NOT TFH_TFDQ_CBB_TABLE (0x1C00, gen2)
    3. SCD_TXFACT (0xA02C10): deactivate all TX DMA FIFOs
    4. SCD_GP_CTRL (0xA02DA8): set AUTO_ACTIVE_MODE (BIT(18))

  Our iwl_dma_init_tfh() had the gen2 TFH_TFDQ_CBB_TABLE writes
  commented out and no gen1 equivalents. No keep-warm buffer, no SCD init.
  The firmware likely crashes when it can't find a valid command queue.
