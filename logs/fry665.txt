fry665 - 2026-03-08 20:43:58 CDT - WiFi 9260: match RFH RX free-ring sizing to Linux 8000/9000 path

SESSION CONTEXT:
  User provided Dell Precision 7530 shell output showing WiFi still stuck at:
    - `wd`: step=9 (alive) rc=-1
    - `ws`: driver=offline
    - `wd2`: RFH_W=0x000000f8, RFH_R=0x00000008, RFH_U=0x00000000
             CPU1_ST=0x00000003, CPU2_ST=0x00002320
             UMAC_PC=0xc0000000, LMAC_PC=0x000000e8

WHY:
  The current failure pattern showed firmware upload succeeding and CPU2 running,
  but RFH still behaving like a 256-entry free-ring path. Linux 7.0-rc2
  `iwlwifi` programs 8000/9000-series MQ RX with a 512-entry RBDCB.

  Our local driver still had:
    - `IWL_RX_QUEUE_SIZE = 256`
    - `RFH_RXF_DMA_RBDCB_SIZE_256`
    - 256 RX data buffers

  That leaves the host-side ring sizing out of sync with what the hardware path
  expects and fits the observed symptom: RFH consumes some free buffers but never
  produces a usable used-ring completion or ALIVE notification.

CHANGES:
  File changed:
    - `src/drivers/net/wifi_9260.c`

  Code updates:
    1. Changed RFH DMA config from 256-entry RBDCB to 512-entry RBDCB.
    2. Increased `IWL_RX_QUEUE_SIZE` from 256 to 512.
    3. Increased RX data-buffer allocation from 256 pages to 512 pages.
    4. Updated RFH init comments / DMA layout comments to match the real ring size.

RATIONALE:
  This is a hardware-agnostic transport fix, not a Dell-specific quirk.
  The Dell was only the machine exposing the mismatch clearly through `wd2`.
