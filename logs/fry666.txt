    - `wd`: shadow_ctrl=0x800fffff, wfpm_gp2_wr=0x01010101,

WHY:
  The newer Dell dump proved the current local boot path was active, firmware
  upload completed, and UMAC had started executing, but LMAC still remained at
  PC=0x000000e8 with no used-ring movement and no ALIVE completion.

  Comparing against Linux 7.0-rc2 `iwlwifi` 8000/9000 transport showed a
  specific mismatch:
    - Per-section progress still uses `FH_UCODE_LOAD_STATUS`
    - The final CPU-ready marker on gen2 uses PRPH
      `UREG_UCODE_LOAD_STATUS` (0xA05C40), not only FH 0x1AF0

  Our driver was writing the final `0xFFFF` / `0xFFFFFFFF` only through
  `FH_UCODE_LOAD_STATUS`. If that register is not the correct final-ready path
  on this part, the CPUs can remain in the exact partially-started state seen
  on Dell.

CHANGES:
  File changed:
    - `src/drivers/net/wifi_9260.c`

  Code updates:
    1. Added `UREG_UCODE_LOAD_STATUS` define for the gen2 final-ready marker.
    2. Added `iwl_write_ucode_load_done()` helper that writes the final marker
       through PRPH instead of only through FH.
    3. Changed runtime CPU image completion to use the new helper for final
       `0xFFFF` / `0xFFFFFFFF` writes.
    4. Cleared both FH and UREG final status paths at upload start so stale
       ready markers do not carry across boots.
    5. Changed the legacy single-path final `0xFFFFFFFF` write to use the same
       gen2-ready helper for consistency.

RATIONALE:
  This is a transport-level fix based on direct Dell hardware evidence and a
  Linux 7.0-rc2 sequencing comparison. It is hardware-agnostic behavior for the
  8000/9000 family, not a Dell-specific quirk.
