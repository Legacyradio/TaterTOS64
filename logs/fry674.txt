fry674 - 2026-03-09 00:27:04 CDT - Shell: remove output truncation, add real scrollbar input, grow scrollback, and fold wd into wdiag

SESSION CONTEXT:
  User called out that the shell itself was making WiFi bring-up painful:
    - long diagnostic output was being cut off
    - the drawn scrollbar did not actually work
    - repeated reboot/test cycles were losing useful context

WHY:
  The bottleneck had moved from the WiFi driver to the shell.
  Reading `src/user/shell/shell.c` showed four concrete issues:
    1. `term_printf()` hard-truncated every formatted print to 512 bytes.
    2. Scrollback was capped at 500 lines and overwrote older output.
    3. The scrollbar was visual-only; mouse events were ignored.
    4. `wdiag` still required a follow-up `wd` request for some failures.

FILES CHANGED:
  - `src/user/shell/shell.c`

CODE CHANGES:
  1. Reworked `term_printf()` to use a two-pass `vsnprintf()` path with
     heap fallback, removing the fixed 512-byte shell print cap.
  2. Replaced fixed 500-line scrollback storage with dynamically growing
     scrollback capacity using `realloc()`.
  3. Added mouse handling for the existing scrollbar so dragging it now
     updates `scroll_offset` instead of doing nothing.
  4. Changed scroll behavior so passive output review stays in place; typing
     a new key snaps back to live output instead of every emitted character.
  5. Raised shell-local `wh` and `wc` buffers from 1024/2048 to 4096 bytes.
  6. Extended `wdiag` to include `wd` before `wl`, `wh`, and `wc`.
  7. Updated shell help text to advertise scrollbar dragging.

RATIONALE:
  This is a shell usability and observability fix, not a Dell-specific hack.
  The goal is to make hardware bring-up evidence retrievable in one boot cycle
  without losing data to shell-side truncation or dead UI affordances.
