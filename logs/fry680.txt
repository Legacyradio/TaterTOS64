fry680 - 2026-03-09 09:03:00 CDT - WiFi: disable unsafe live reset command that can hard-freeze the machine

SESSION CONTEXT:
  The user reported that running `wifi reset` froze the Dell Precision 7530 so
  completely that the uptime counter stopped moving. This was a hard machine
  hang, not a recoverable shell/UI failure.

WHY:
  The current `wifi reset` path was not a real reset/recovery path. It invoked
  `wifi_9260_init()` again on a live system without a matching teardown for the
  existing WiFi state, IRQ/MSI setup, or other driver-owned resources. Exposing
  that as an operator command was unsafe.

FILES CHANGED:
  - `src/user/shell/shell.c`
  - `src/drivers/net/wifi_9260.c`
  - `logs/fry680.txt`

CODE CHANGES:
  1. Stopped advertising `wifi reset` in shell help / command listings.
  2. Changed the shell handler to report that live reset is disabled and that a
     reboot is required until a real teardown/recovery path exists.
  3. Changed `wifi_9260_reinit_user()` to reject userspace live reinit with
     `-95` instead of attempting a second full initialization.

RATIONALE:
  A disabled command is better than a command that can wedge the whole machine.
  A future re-enable must follow the same pattern as the guarded NVMe reset
  path: explicit teardown, IRQ/MSI cleanup, recursion guards, and bounded
  recovery.
