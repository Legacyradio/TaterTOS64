fry681 - 2026-03-09 09:18:00 CDT - WiFi 9260: restore Linux-matching CPU2 secure-header pointer in direct FH upload path

SESSION CONTEXT:
  The current Dell Precision 7530 failure shape remained:
    - init stalls at step 9 (`alive`)
    - `UMAC_PC` runs but `LMAC_PC` stays pinned at `0x000000e8`
    - `RFH_R` / `RFH_U` remain zero

  After `fry678`, the driver no longer wrote any secure-header PRPH register in
  the direct `_8000` upload path. That removed the bogus old writes, but it also
  removed the one Linux-matching CPU2 header pointer write that still exists in
  the reference path.

WHY:
  Linux 7.0-rc2 `iwlwifi` `iwl_pcie_load_given_ucode_8000()` still programs:
    - `LMPM_SECURE_UCODE_LOAD_CPU2_HDR_ADDR = LMPM_SECURE_CPU2_HDR_MEM_SPACE`

  before loading CPU2 sections in the `_8000` direct-FH flow. The current local
  driver was only observing those PRPH registers, not restoring the required
  CPU2 pointer at the corrected PRPH offset (`0x1E7C`). That leaves CPU2/LMAC
  bring-up as the strongest remaining transport mismatch for the observed stall.

FILES CHANGED:
  - `src/drivers/net/wifi_9260.c`
  - `logs/fry681.txt`

CODE CHANGES:
  1. In the secure multi-section runtime upload path:

       `LMPM_SECURE_UCODE_LOAD_CPU2_HDR_ADDR = 0x420400`
  2. Read the register back and log the value for the next hardware dump.
  3. Left the earlier bogus register definitions/writes dead:
     - no return to the wrong `0xA01000` / `0xA01100` offsets
     - no blanket reintroduction of both header writes

  remaining CPU2/LMAC stall signature.
