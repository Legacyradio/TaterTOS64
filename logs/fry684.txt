  The user-provided Dell Precision 7530 bare-metal dump showed the most
  progressed known failure shape so far:
    - `step=9 rc=-1 irq=2`
    - `FH_LOAD=0xffffffff`
    - `UREG_UCODE_LOAD_STATUS=0x00000000`
    - `CPU1_ST=0x00000003`
    - `CPU2_ST=0x00002320`
    - `UMAC_PC=0xc0000000`
    - `LMAC_PC=0x000000e8`
    - no RX progress (`RFH_R=0`, `RFH_U=0`)

WHY:
  In Linux 7.0-rc2, the same direct upload code used for 8000/9000-class parts
  still keeps `FH_UCODE_LOAD_STATUS` for the per-section bitmap, but for
  `trans->mac_cfg->gen2` it writes the final CPU-ready marker to
  `UREG_UCODE_LOAD_STATUS`, not `FH_UCODE_LOAD_STATUS`.

  The local driver was mixing those semantics:
    - section progress bits were written into `FH_UCODE_LOAD_STATUS`
    - final completion was also written into `FH_UCODE_LOAD_STATUS`
    - `UREG_UCODE_LOAD_STATUS` remained zero in the live Dell dump

  Since the user's current best state is the progressed `step=9 irq=2` shape,
  this change intentionally stays on that same direct `_8000` path and only
  corrects the final done-marker register inside that path.
