fry685 - 2026-03-09 21:08:16 CDT - WiFi 9260: re-doorbell RFH RX free-ring at ALIVE handoff

SESSION CONTEXT:
  The latest Dell Precision 7530 bare-metal dump had moved past firmware upload
  and CPU release, but still stalled at step 9 (`alive`) with:
    - `FH_LOAD=0x000f000f`
    - `CPU1_ST=0x030a0003`
    - `CPU2_ST=0x000f000f`
    - `UMAC_PC=0xc0800000`
    - `LMAC_PC=0x000605dc`
    - `RFH_ACT=0x00010001`
    - `RFH_W=0x00000000`, `RFH_R=0x00000000`, `RFH_U=0x00000000`
    - `rx_closed=0x00000000`, `rx_fin=0x00000000`

WHY:
  The dump no longer matched the older "firmware never started" or bad secure
  header cases. The strongest remaining host-side theory was that the RX
  free-ring advertisement was getting lost at the boundary between firmware
  upload completion and the ALIVE wait.

  The RX ring base addresses were already matching the programmed DMA values,
  but RFH still showed zero free-ring consumption and zero used/status writes.
  Re-advertising the free-ring at the ALIVE boundary is hardware-agnostic and
  directly targets that remaining handoff gap without changing the proven
  upload ordering.

FILES CHANGED:
  - `src/drivers/net/wifi_9260.c`

CODE CHANGES:
  1. Added `iwl_rfh_kick_rx_free_ring()` in `src/drivers/net/wifi_9260.c`.
     - Flushes RXBD / RXUSED / RX status memory.
     - Rewrites `RFH_Q_FRBDCB_WIDX_TRG(0)` with the full rounded-down queue
       advertisement.
     - Captures an optional boot-trace tag.

  2. Switched initial RFH queue advertisement to use that helper.
     - keeps the init-time doorbell path and the ALIVE-time doorbell path
       using the same logic.

  3. Re-doorbell RFH at ALIVE wait start.
     - Immediately after rotating interrupts to `ALIVE | FH_RX`, call the
       helper again before polling for notifications.

  4. Added one bounded RFH re-kick during ALIVE timeout polling.
     - If there is still zero RX status progress after 50 ms, re-advertise the
       free-ring one more time and trace it as `alive-rekick`.

RATIONALE:
  This change is deliberately narrow. It does not revisit secure-header,
  context-info, or upload-completion logic because the user's latest hardware
  dump already showed those paths progressing much further than before.
