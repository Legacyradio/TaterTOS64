fry686 - 2026-03-09 22:28:04 CDT - WiFi 9260: fix CPU-done marker register (FH vs UREG)

SESSION CONTEXT:
  fry685 added an RFH RX free-ring re-doorbell at the ALIVE handoff boundary.
  Hardware test on Dell Precision 7530 showed:
    - `UMAC_PC` shifted from `0xc0800000` to `0xc0000000` (firmware relocated)
    - `LMAC_PC` frozen at `0x000605dc` across both snapshots
    - `RFH_W=0`, `RFH_R=0`, `RFH_U=0` -- RFH never consumed anything
    - `UREG_UCODE_LOAD_STATUS=0x00000000` -- firmware never acknowledged start
    - `LMPM_SECURE_CPU1_HDR=0x00000000`, `LMPM_SECURE_CPU2_HDR=0x00000000`
    - `CPU1_ST=0x030a0003`, `CPU2_ST=0x000f000f`
    - `FH_LOAD=0x000f000f` -- per-section bits present but final done missing
    - Boot trace confirmed `alive-kick` and `alive-rekick` both fired with no
      RX movement.  Firmware never sent ALIVE.

  Full wd3 phase trace showed the upload sequence completing correctly
  (sections loaded, per-section FH bits accumulating) but firmware stalling
  immediately after -- LMAC never started, UMAC stuck at early init.

ROOT CAUSE:
  `iwl_write_ucode_load_done()` was writing the final CPU-done marker to
  `UREG_UCODE_LOAD_STATUS` (PRPH register `0xA05C40`) via `iwl_write_prph()`.

  The 9260 (device 0x2526) is 9000-series, which is NOT gen2.  In the Linux
  iwlwifi driver (`pcie/gen1_2/trans.c`, `iwl_pcie_load_cpu_sections_8000()`),
  the done marker destination depends on `trans->mac_cfg->gen2`:
    - gen2 = true  -> `iwl_write_prph(UREG_UCODE_LOAD_STATUS, 0xFFFF/0xFFFFFFFF)`
    - gen2 = false -> `iwl_write_direct32(FH_UCODE_LOAD_STATUS, 0xFFFF/0xFFFFFFFF)`

  `iwl9000_mac_cfg` in `cfg/9000.c` does NOT set `gen2 = true`, so Linux uses
  `FH_UCODE_LOAD_STATUS` (CSR register at `0x1AF0`) for the done marker.

  The firmware boot stub on 9260 polls `FH_UCODE_LOAD_STATUS` for `0xFFFF`
  (CPU1 done) and `0xFFFFFFFF` (CPU2 done).  Because TaterTOS was writing the
  done marker to the wrong register, `FH_UCODE_LOAD_STATUS` stayed at
  `0x000f000f` (per-section bits only), and firmware never saw the "all
  sections ready" signal.

  This explains why:
    - `UMAC_PC` relocated to `0xc0000000` (UMAC started but stalled waiting
      for the done marker)
    - `LMAC_PC` stuck at `0x000605dc` (UMAC never released LMAC)
    - `UREG_UCODE_LOAD_STATUS=0x00000000` (firmware never wrote this register
      because it never reached that stage)
    - RFH was irrelevant -- firmware never got far enough to touch RX DMA

  The per-section bitmap writes in `iwl_mark_cpu_section_loaded()` were already
  correctly going to `FH_UCODE_LOAD_STATUS` via `iwl_write32()`.  Only the
  final done marker was misdirected.
