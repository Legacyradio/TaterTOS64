    - `src/drivers/net/wifi_9260.c`
    - `iwl_upload_runtime_cpu_image()`
    - old behavior:
        `if (loaded == 0) return -11;`

  That behavior is acceptable for runtime SEC_RT splitting when the section
  stream is separator-delimited and a missing second CPU image likely means the
  parser/split logic is wrong.

  It is not Core43-consistent for the current INIT path:
    - `eventually4` records that SEC_INIT in this blob is flat, not
      separator-delimited
    - the current two-phase code reuses the same uploader for INIT and runtime
    - on INIT:
        CPU1 can legitimately consume the flat SEC_INIT stream
        CPU2 can then start at the end of the array
        a local `loaded == 0` hard fail becomes `step=8 rc=-11`

  Local upstream comparison supports relaxing that rule:
    - `linux-7.0-rc2/.../pcie/gen1_2/trans.c`
    - `iwl_pcie_load_cpu_sections_8000()`
    - the Linux path does not fail merely because the second pass copies zero
      sections; it still enables interrupts and writes the CPU-done marker

  This makes the current `step=8 rc=-11` a local regression in upload
  semantics, not proof that the firmware rejected the INIT phase.
