WHY THIS PATCH:
  The current source still mixed protocol models on the ALIVE path.

  Specifically, local code still treated ALIVE as:
    - command/group pair:
      - `group = 0xff`
      - `cmd   = 0xff`

  That does not match the Core43 / 9000-family MVM model used by this card.

  Local upstream reference in the bundled Linux tree says:
    - file:
      `linux-7.0-rc2/drivers/net/wireless/intel/iwlwifi/fw/api/commands.h`
    - command groups:
      - `LEGACY_GROUP = 0x0`
      - `SYSTEM_GROUP = 0x2`
    - legacy command IDs:
      - `UCODE_ALIVE_NTFY = 0x1`
      - `INIT_COMPLETE_NOTIF = 0x4`

  That means:
    - ALIVE should be received as:
      - `group = 0x00`
      - `cmd   = 0x01`
    - not `0xff/0xff`

  This was already called out as one of the highest-confidence issues in:
    - `eventually/eventually4.txt`
    - diagnosis section:
      "wrong ALIVE / INIT_COMPLETE IDs and group semantics"

  Important clarification discovered during this step:
    - `eventually4` contains one stale sub-detail when it says current source
      still waits for `INIT_COMPLETE = 0x84`
    - current `wifi_9260.c` was already corrected to:
      - `IWL_INIT_COMPLETE_NOTIF = 0x04`
    - so the concrete remaining mismatch in current source was ALIVE semantics,
      not INIT_COMPLETE

EXACT CODE CHANGES:
  File:
    - `src/drivers/net/wifi_9260.c`

  1. Constant block update in ALIVE-handshake section
     Current line range after patch:
       - `src/drivers/net/wifi_9260.c:2152-2161`

     Old values:
       - `IWL_ALIVE_CMD_ID  = 0xFF`
       - `IWL_SYSTEM_GROUP  = 0xFF`
       - `IWL_MVM_ALIVE_CMD = 0xFF`

     New values:
       - `IWL_LEGACY_GROUP  = 0x00`
       - `IWL_SYSTEM_GROUP  = 0x02`
       - `IWL_ALIVE_CMD_ID  = 0x01`
       - `IWL_MVM_ALIVE_CMD = IWL_ALIVE_CMD_ID`

     Rationale for each:
       - `IWL_LEGACY_GROUP = 0x00`
         because Core43/MVM ALIVE is a legacy-group notification
       - `IWL_SYSTEM_GROUP = 0x02`
         corrected to match upstream command-group definitions even though this
         particular patch does not yet consume `SYSTEM_GROUP` on the ALIVE path
       - `IWL_ALIVE_CMD_ID = 0x01`
         because upstream defines `UCODE_ALIVE_NTFY = 0x1`
       - `IWL_MVM_ALIVE_CMD = IWL_ALIVE_CMD_ID`
         removes duplicate divergent meaning for the same notification

  2. ALIVE wait-path RX filter update
     Current line:
       - `src/drivers/net/wifi_9260.c:2664`

     Old logic:
       - accepted ALIVE only if:
         - `group == IWL_SYSTEM_GROUP`
         - `cmd == IWL_ALIVE_CMD_ID`
       - with old constants, that reduced to `0xff/0xff`

     New logic:
       - accepts ALIVE only if:
         - `group == IWL_LEGACY_GROUP`
         - `cmd == IWL_ALIVE_CMD_ID`
       - this now means `0x00/0x01`

     Why this matters:
       - `iwl_wait_alive()` is the authoritative bring-up gate
       - if this filter waits for the wrong notification ID/group, the driver
         can time out in step 9 even if the RX transport is otherwise correct

  3. Generic RX classifier ALIVE check update
     Current line:
       - `src/drivers/net/wifi_9260.c:3604`

     Old logic:
       - `if (group == 0xFF && cmd == 0xFF) return 1;`

     New logic:
       - `if (group == IWL_LEGACY_GROUP && cmd == IWL_ALIVE_CMD_ID) return 1;`

     Why this matters:
       - this keeps the secondary RX classification path consistent with the
         explicit ALIVE wait path
       - avoids one path being fixed while another still silently assumes
         `0xff/0xff`
