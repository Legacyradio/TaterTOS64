SESSION CONTEXT:
  The active repo state after `fry700` already exposed:
    - upload-pass summaries in:
      - `wd3`
      - `wc`
      - `wh`
    - fixed live SRAM probes only in:
      - `wd2`
    - the UMAC error table only in:
      - `wc`

  The user then explicitly asked for:
    - one command
    - named `sram`
    - with "all sram info we could ever need"

  That request mattered because the current tree still had no single command
  that showed, in one place:
    - which SRAM destinations were actually staged by the active uploader
    - what bytes the host expected at those destinations
    - what readback looked like immediately after upload
    - what the same addresses look like when inspected later from the shell

  In other words, fry700 had already made the upload sequencing visible, but
  the SRAM evidence itself was still fragmented.

WHY THIS PATCH:
  After fry700, the source truth was:
    - `wd2` could show a few fixed live probes:
      - `0x00000000`
      - `0x00404000`
      - `0x00800000`
      - `0x00800020`
    - `wc` could show the UMAC error table at `0x00800000`
    - `wd3` could show upload-pass metadata
    - but none of those commands retained section-by-section SRAM readback for
      the actual image that had just been staged

  That gap matters most on the current branch because `eventually12` had
  already narrowed the lead to:
    - the deeper firmware-precondition / section-boundary / assert side
    - not the already-spent ALIVE restock branch

  So the next practical diagnostic step was not another RFH or ALIVE tweak.
  It was:
    - a single shell command that exposes the SRAM view needed to judge the
      currently staged section stream directly

  This patch intentionally does introduce a new shell command.
  That differs from fry700 on purpose because this step was driven by an
  explicit user request and by the still-split SRAM evidence surface.
