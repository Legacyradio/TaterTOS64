    - `size = payload size only`

WHY THIS PATCH:
  After the `eventually12` / `eventually13` narrowing, the project had
  already spent:
    - ALIVE-side restock timing
    - interrupt-mask timing around done markers
    - RFH visibility and queue-state checks
    - SRAM observability improvements

  The remaining serious host-side mismatch was narrower:
    - secure TLV parse semantics
    - secure upload semantics
    - separator handling under those semantics
    - context-info payload copying under those semantics

  Linux stores secure sections as:
    - `offset = parsed from the TLV header`
    - `data = payload after the offset`
    - `size = payload length after the offset`

  The local tree did not.

  That mismatch is exactly the kind of defect that can survive many hardware
  logs while still being a real bring-up blocker.

  So fry702 rewrites the secure-section contract end-to-end instead of making
  another local band-aid around the old mixed model.

EXACT CODE CHANGES:
  File:
    - `src/drivers/net/iwlwifi_fw.c`

  1. Added a dedicated secure-section store helper and changed secure TLV
     parsing to Linux-style payload-only semantics.
     Relevant lines after patch:
       - `164-181`
       - `270-290`

     New behavior:
       - secure sections now store:
         - `offset = fw_u32le(tlv_data)`
         - `data = tlv_data + 4`
         - `size = tlv_len - 4`

     Why this matters:
       - the parser no longer leaves the 4-byte destination header inside
         `sec->data / sec->size`

  2. Added support for secure secure-section TLVs.
     Relevant lines after patch:
       - `270-290`

     New behavior:
       - `SECURE_SEC_RT`
       - `SECURE_SEC_INIT`
       are now routed through the same secure-section store helper

     Why this matters:
       - the secure-section path now matches Linux more honestly even if a
         different blob family is used later

  File:
    - `src/drivers/net/iwlwifi_fw.h`

  3. Clarified the public section contract.
     Relevant lines after patch:
       - `62-65`

     New behavior:
       - `struct iwl_fw_section` is now explicitly documented as:
         - payload-only `data`
         - payload-only `size`
         - separate `offset`

     Why this matters:
       - the parser / transport agreement is now explicit, not implied

  File:
    - `src/drivers/net/wifi_9260.c`

  4. Replaced the old mixed "load section by reparsing its first 4 bytes"
     helper with an explicit payload uploader.
     Relevant lines after patch:
       - `1918-1982`

     New behavior:
       - uploads now take:
         - explicit SRAM destination
         - payload pointer
         - payload size
       - the transport does not reparse an offset header from secure payload
         bytes anymore

     Why this matters:
       - the uploader now consumes the same section contract the parser
         produces

  5. Reworked the CPU1 / CPU2 secure upload loop to operate on stored offsets
     and Linux-style separator handling.
     Relevant lines after patch:
       - `2000-2086`

     New behavior:
       - each real secure section loads from:
         - `iwl_fw_sec_get_offset(sec)`
         - `sec->data`
         - `sec->size`
       - separator handling no longer depends on the 4-byte header still being
         embedded in the payload
       - zero-length real payloads now fail explicitly instead of being
         silently treated like the old raw-header case

     Why this matters:
       - CPU pass iteration now matches the Linux `_8000` mental model much
         more closely

  6. Fixed secure-section helper semantics for:
       - offset lookup
       - separator detection
       - runtime section counting
       - context-info DMA copying
     Relevant lines after patch:
       - `2239-2289`

     New behavior:
       - separator detection keys off stored section offsets directly
       - section counting no longer assumes `size >= 4` for separators
       - DMA copying now copies the stored payload exactly, not `data + 4`

     Why this matters:
       - the whole secure-section stack now uses one consistent contract
