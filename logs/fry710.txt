SECTION 1:  9000-SPECIFIC BUGS WE ARE MISSING  (HIGH PRIORITY)
===========================================================================

--- BUG A: PERSISTENCE BIT NOT CLEARED (9000-specific) ---

Linux file: pcie/gen1_2/trans.c line 1734-1762
Function:   iwl_trans_pcie_clear_persistence_bit()
Called from: _iwl_trans_pcie_start_hw() at line 1799
When:        BEFORE sw_reset, BEFORE apm_init — very early in boot

Linux code (paraphrased):
  if (device_family == IWL_DEVICE_FAMILY_9000) {
      wprot = PREG_PRPH_WPROT_9000;  // 0xA04CE0
  }
  hpm = iwl_read_umac_prph(HPM_DEBUG);  // prph 0xA03440
  if (hpm & PERSISTENCE_BIT) {          // BIT(12) = 0x1000
      wprot_val = iwl_read_umac_prph(wprot);
      if (wprot_val & PREG_WFPM_ACCESS) {
          ERROR "can not clear persistence bit";
          return -EPERM;
      }
      iwl_write_umac_prph(HPM_DEBUG, hpm & ~PERSISTENCE_BIT);
  }

Our code: NOTHING.  We never read HPM_DEBUG, never clear PERSISTENCE_BIT.

Why this matters:
  The persistence bit preserves power management state across resets.
  If UEFI (Dell Precision 7530 UEFI firmware) or a previous boot left
  this bit set, the 9260 firmware sees stale power management state
  during init.  The firmware's early self-check may ASSERT because the
  hardware state doesn't match what a clean boot should look like.

  This is ONLY done for DEVICE_FAMILY_9000 and DEVICE_FAMILY_22000.
  Not for 8000 or older.  By treating the 9260 as generic "8000+" we
  completely missed this.

Register addresses:
  HPM_DEBUG          = 0xA03440 (UMAC peripheral register)
  PERSISTENCE_BIT    = BIT(12) = 0x00001000
  PREG_PRPH_WPROT_9000 = 0xA04CE0 (write protection check)
  PREG_WFPM_ACCESS   = (need to look up exact bit, but it's a guard)


--- BUG B: CSR_GIO_CHICKEN_BITS DIS_L0S_EXIT_TIMER (8000+ skip) ---

Linux file: pcie/gen1_2/trans.c line 243-246
Function:   iwl_pcie_apm_init()

Linux code:
  if (trans->mac_cfg->device_family < IWL_DEVICE_FAMILY_8000)
      iwl_set_bit(CSR_GIO_CHICKEN_BITS, DIS_L0S_EXIT_TIMER);  // BIT(29)

Our code (wifi_9260.c line 1026-1027):
  iwl_set_bit(CSR_GIO_CHICKEN_BITS, DIS_L0S_EXIT_TIMER);  // ALWAYS

The 9260 is FAMILY_9000 which is >= FAMILY_8000, so Linux SKIPS this.
We write it unconditionally.

NOTE: fry709 already addressed this partially by making DIS_L0S_EXIT_TIMER
a fallback rather than default.  But fry709 still attempts it on retry.
The correct behavior is to NEVER set it for 9000-series.

Why this matters:
  DIS_L0S_EXIT_TIMER disables the L0s power state exit timer on the
  PCIe link.  For 8000+ devices, Intel changed the L0s handling and
  this timer disable can interfere with PCIe link state transitions.
  Setting it on a 9000-series device may cause the PCIe link to
  behave incorrectly during DMA transfers, potentially corrupting
  the firmware's ALIVE notification DMA or causing bus errors that
  trigger SYSASSERT.

Register: CSR_GIO_CHICKEN_BITS = 0x100, bit 29 = 0x20000000


--- BUG C: RFKILL_WAKE_L1A_EN NOT SET (9000+ specific) ---

Linux file: pcie/gen1_2/internal.h line 1035-1043
Function:   iwl_enable_rfkill_int()

Linux code:
  if (trans->mac_cfg->device_family >= IWL_DEVICE_FAMILY_9000) {
      iwl_set_bit(CSR_GP_CNTRL, CSR_GP_CNTRL_REG_FLAG_RFKILL_WAKE_L1A_EN);
  }

Our code: We never set RFKILL_WAKE_L1A_EN.

CSR_GP_CNTRL_REG_FLAG_RFKILL_WAKE_L1A_EN = 0x04000000 = BIT(26)

Why this matters:
  Comment in Linux says "On 9000-series devices this bit isn't enabled
  by default, so when we power down the device we need set the bit to
  allow it to wake up the PCI-E bus for RF-kill interrupts."

  Without this bit, the device may not properly handle RF-kill state
  transitions.  While this alone probably doesn't cause SYSASSERT,
  it's part of the 9000-specific init that we're skipping entirely.

  Called in iwl_enable_rfkill_int() which runs during _start_hw,
  BEFORE firmware upload.

SECTION 2:  INIT SEQUENCE ORDERING DIFFERENCES  (MEDIUM PRIORITY)

--- DIFFERENCE D: Linux calls nic_config BEFORE RX/TX init ---

Linux order (in iwl_pcie_nic_init, trans.c ~line 485-520):
  1. iwl_pcie_apm_init()          → APM clock + power
  2. iwl_op_mode_nic_config()     → CSR_HW_IF_CONFIG_REG (radio type, SI bits)
  3. iwl_pcie_rx_init()           → RFH setup, RX queues
  4. iwl_pcie_tx_init()           → FH channels, SCD, KW buffer

Our order (in iwl_fw_boot_attempt, wifi_9260.c ~line 2850-2960):
  1. iwl_nic_reset()              → APM + MAC access
  2. iwl_dma_alloc()              → buffer allocation
  3. iwl_dma_init_tfh()           → TX + RX combined
  4. iwl_nic_config()             → CSR_HW_IF_CONFIG_REG ← AFTER RX/TX!

Our nic_config is called AFTER RFH init.  Linux calls it BEFORE.
If the RFH hardware or the firmware reads CSR_HW_IF_CONFIG_REG during
RFH bring-up, it would find unconfigured values in our case.

The SI (step/integration) bits in CSR_HW_IF_CONFIG_REG tell the hardware
what radio/PHY configuration to use.  If these are wrong during RFH
DMA setup, the hardware may misconfigure the DMA path.


--- DIFFERENCE E: Linux calls APM init TWICE ---

Linux boot flow has TWO phases:
  Phase 1: _iwl_trans_pcie_start_hw()
    - prepare_card_hw
    - clear_persistence_bit        ← 9000-specific
    - sw_reset(retake_ownership)   ← includes prepare_card_hw again
    - apm_init                     ← FIRST APM init
    - init_msix
    - enable_rfkill_int            ← 9000-specific L1A bit

  Phase 2: iwl_trans_pcie_start_fw()
    - iwl_pcie_nic_init():
      - apm_init                   ← SECOND APM init
      - nic_config
      - rx_init
      - tx_init
    - shadow_reg_enable
    - fw_load_int
    - clear_rfkill
    - upload_firmware
    - wait_alive

Our boot flow has ONE phase:
  iwl_fw_boot_attempt():
    - iwl_nic_reset (includes apm_init ONCE, no persistence bit clear)
    - iwl_dma_alloc
    - iwl_dma_init_tfh
    - iwl_nic_config
    - iwl_enable_msi
    - clear rfkill
    - clear interrupts
    - shadow_reg
    - fw_load_int
    - clear rfkill again
    - upload_firmware
    - wait_alive

We skip the entire Phase 1.  Linux's Phase 1 does persistence bit
clearing, an extra SW reset + prepare_card_hw, and RFKILL L1A enable.


--- DIFFERENCE F: Missing sleep after SW_RESET ---

Linux's iwl_trans_pcie_sw_reset() (trans.c line 78-95):
  For non-BZ family (i.e., 9260):
    iwl_set_bit(CSR_RESET, CSR_RESET_REG_FLAG_SW_RESET);
    usleep_range(5000, 6000);   ← 5-6ms sleep!
    if (retake_ownership)
        return iwl_pcie_prepare_card_hw();

Our iwl_sw_reset() (wifi_9260.c line 1065-1066):
  iwl_set_bit(CSR_RESET, CSR_RESET_REG_FLAG_SW_RESET);
  // NO SLEEP — returns immediately!

The hardware may need 5ms to complete the software reset.  Proceeding
immediately could mean we're configuring the NIC before it has fully
reset, leading to undefined register states.

===========================================================================

User mentioned "SWM_CHICKEN" at 0xCA0 with bits 1 and 4 for DMA reset.

This register name does NOT appear in the Linux iwlwifi driver source.

The iwlwifi driver does have these "chicken bit" registers:

from a non-public Intel document, it doesn't apply to iwlwifi.
However, there may be undocumented registers not in the open-source
driver — would need Intel datasheet to confirm.

SECTION 11:  WHAT TO FIX IN ORDER OF PRIORITY
===========================================================================

FIX 1 (HIGHEST PRIORITY — 9000-specific):
  Add persistence bit clearing BEFORE sw_reset in iwl_nic_reset().
  Read HPM_DEBUG (prph 0xA03440), if BIT(12) set, check write-protection
  at PREG_PRPH_WPROT_9000 (prph 0xA04CE0), then clear BIT(12).
  Requires grab_nic_access to read/write UMAC prph.

FIX 2 (HIGH PRIORITY — wrong for 8000+):
  Remove DIS_L0S_EXIT_TIMER write in iwl_apm_init() entirely.
  The 9260 is family 9000, Linux NEVER sets this for >= 8000.
  fry709 made it a fallback, but it should be REMOVED completely.
  Only set L1A_NO_L0S_RX (which IS set for all families).

FIX 3 (HIGH PRIORITY — 9000-specific):
  Set RFKILL_WAKE_L1A_EN in CSR_GP_CNTRL.
  CSR_GP_CNTRL |= 0x04000000 (BIT(26)).
  Do this after enabling rfkill interrupt, before firmware upload.

FIX 4 (MEDIUM PRIORITY — missing sleep):
  Add 5ms sleep after sw_reset before continuing.
  Linux's iwl_trans_pcie_sw_reset() does usleep_range(5000, 6000).
  Our iwl_sw_reset() (line 1065-1066) just sets the bit and returns.
  The hardware may need time to complete the reset.

FIX 5 (MEDIUM PRIORITY — ordering):
  Move iwl_nic_config() call to BEFORE iwl_dma_init_tfh().
  Match Linux's order: apm_init → nic_config → rx_init → tx_init.

FIX 6 (LOW PRIORITY — post-ALIVE, for later):
  After ALIVE: enable FH TX DMA channels 0-9 with
  FH_TCSR_TX_CONFIG_REG = DMA_CHNL_ENABLE | DMA_CREDIT_ENABLE.
  Set FH_TX_CHICKEN_BITS_SCD_AUTO_RETRY_EN.

SECTION 12:  WHY "8000+" WAS WRONG
===========================================================================

The user correctly identified that we've been treating the 9260 as
generic "8000+ series" everywhere.  The firmware UPLOAD path is indeed
shared (iwl_pcie_load_given_ucode_8000 is used by both 8000 and 9000).
But the SURROUNDING INIT has 9000-specific differences:

1. Persistence bit clearing — ONLY for 9000 and 22000, not 8000
2. RFKILL_WAKE_L1A_EN — ONLY for >= 9000, not 8000
3. DIS_L0S_EXIT_TIMER — skipped for >= 8000 but we didn't skip it
4. PREG_PRPH_WPROT uses a 9000-specific address
5. MSI-X max RX queues has a 9000-specific limit

By focusing exclusively on the 8000+ upload path, we missed all the
9000-specific initialization that Linux does in the surrounding code.

The SYSASSERT has been the same since fry661 — meaning the root cause
has NEVER been in the upload path.  It's in the init that SURROUNDS
the upload.  The persistence bit and L0s timer are the two most likely
culprits because they affect DMA and power state at boot time.
