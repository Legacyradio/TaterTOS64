ROOT CAUSE OF fry718 REGRESSION:
  fry718 made two changes based on incorrect analysis of Linux's code:

  1. Skipped RELEASE_CPU_RESET for single-CPU INIT images
  2. Skipped CPU2 call for single-CPU INIT images

  BUT: Linux's iwl_pcie_load_given_ucode_8000() UNCONDITIONALLY calls
  CPU2 — there is NO is_dual_cpus check.  For single-CPU INIT, the CPU2
  call loads 0 sections but STILL writes 0xFFFFFFFF to FH_UCODE_LOAD_STATUS.
  This is the final "go" signal that tells the boot ROM to start firmware.

  By skipping the CPU2 call, fry718 left the done marker at 0x0000FFFF.
  The boot ROM never received the "all CPUs ready" signal, firmware never
  started, DMA results were never committed to SRAM.

  fry718's sentinel removal (change 4) was correct and is preserved.

CHANGES:

  1. Restore unconditional RELEASE_CPU_RESET
     File: src/drivers/net/wifi_9260.c, iwl_pcie_load_given_ucode_8000()
     Why: fry718 made this conditional on has_cpu2.  Restored to always
     write, matching fry716 behavior that got firmware running.

  2. Restore unconditional CPU2 call
     File: src/drivers/net/wifi_9260.c, iwl_pcie_load_given_ucode_8000()
     Why: Linux ALWAYS calls iwl_pcie_load_cpu_sections_8000(cpu=2).
     For single-CPU INIT, it loads 0 sections but writes the critical
     0xFFFFFFFF done marker.  fry718 skipped this entirely.

  Sentinel writes remain removed (fry718 change 4 — section[1] corruption fix).
  SRAM checkpoint diagnostics from fry718 remain in place.
