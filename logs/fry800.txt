fry800 — Multi-host connections, CSS @import, TLS hardening, recv fixes
=====================================================================
Timestamp: 2026-04-03

SESSION CHANGES:

1. MULTI-HOST CONNECTIONS — tatersurf.c, ts_resource.h
   - Cross-host HTTPS resources now use independent BearSSL connections
     per resource slot (4 concurrent). Previously avoided due to mbedTLS
     crash (now irrelevant with BearSSL switch in fry798-799).
   - Updated outdated mbedTLS crash comments throughout
   - Added CROSS-HOST: serial debug logging in ENQUEUE_RES macro
   - Resource manager already had the infrastructure; just needed
     comment cleanup and verification

2. CSS @IMPORT SUPPORT — ts_css.h, tatersurf.c
   - New ts_css_extract_imports() in ts_css.h — parses @import url("..."),
     @import "...", @import '...' at top of CSS files
   - Handles @charset, CSS comments before @import rules
   - Up to 8 @import URLs per CSS file
   - Wired into both reuse_process_completion and process_resource_completions
   - Resolves @import URLs relative to CSS file URL (not page URL)
   - Enqueues discovered imports as TS_RES_CSS resources

3. TLS HARDENING — ts_http.h ts_tls_setup()
   - Restricted cipher suites to ECDHE-only + AEAD-only:
     * ECDHE_ECDSA/RSA + AES-256-GCM-SHA384 (preferred)
     * ECDHE_ECDSA/RSA + CHACHA20-POLY1305-SHA256
     * ECDHE_ECDSA/RSA + AES-128-GCM-SHA256
   - Enforced TLS 1.2 minimum (no TLS 1.0/1.1 fallback)
   - Excluded: all RSA key exchange (no forward secrecy),
     all CBC ciphers (padding oracle), 3DES, RC4, NULL, export
   - Still uses br_ssl_client_init_full for crypto impl registration,
     then overrides suite list and version range

4. NON-BLOCKING HTTP SOCKETS — ts_http.h ts_http_get()
   - Moved fry_fcntl(O_NONBLOCK) before the HTTPS/HTTP branch so ALL
     sockets are non-blocking, not just HTTPS
   - ts_http__recv for plain HTTP now returns -2 (try again) instead of
     -1 (fatal error) when fry_recv returns EAGAIN
   - Fixes: HTTP redirect targets (e.g. youtube.com:80 → https) were
     blocking the event loop or erroring out immediately

5. BEARSSL RECV PUMP LOOP — ts_http.h ts_http__recv()
   - Changed single-pump to loop (up to 32 iterations)
   - Pumps socket→BearSSL until RECVAPP available or socket empty
   - Fixes: partial TLS records spanning multiple socket reads were
     never fully assembled in a single poll cycle

6. BODY DATA STUCK IN RECV_BUF — ts_http.h ts_http_poll()
   - Added early check: if headers_complete and recv_pos < recv_len,
     jump to process_body before attempting new recv
   - ROOT CAUSE: after header parsing, leftover body data sat in
     recv_buf. Next poll call got n=-2 (no NEW data) and bailed
     at the -2 check, never processing the existing body data.
   - This was THE bug blocking YouTube and all non-Google sites.

7. IMAGE CACHE PRESERVATION — tatersurf.c reuse_process_completion(),
   process_resource_completions()
   - Save/restore img_cache across ts_doc_init() relayout calls
   - Previously ts_doc_init() zeroed the entire document including
     decoded image pixel data, destroying fetched images after relayout

8. STATUS BAR FIX — tatersurf.c reuse_process_completion()
   - Added set_status("Done") + needs_redraw after reuse queue finishes
     and g_resources_loading becomes 0
   - Previously status bar showed stale "Loading resources... (0 queued)"
     because the "Done" path only ran inside the resources_loading block
     which was already false

9. PAGE DEBUG LOGGING — tatersurf.c build_page()
   - Added: PAGE: nodes=N links=N css=N js=N img=N after layout
   - Added: HIT: vx=N vy=N link=N/N on viewport clicks
   - Added: POLL: state=N n=N hdr=N body=N fd=N https=N (every 100th call)
   - Added: REDIRECT[N]: status -> url in ts_http_follow_redirect

10. HTTPS-FIRST DEFAULT — tatersurf.c navigate()
    - Bare URLs (no scheme) now default to https:// instead of http://
    - Explicit http:// still respected for sites that need it

BUG FIXES DURING SESSION (additional):

11. KERNEL TCP RECV DRAIN — src/kernel/proc/syscall.c SYS_RECV
    - ROOT CAUSE of "recv failed: -1": kernel checked sk->state !=
      SOCK_ST_CONNECTED BEFORE calling tcp_recv(). When server sent
      FIN (connection close), kernel returned -ENOTCONN (107) without
      draining buffered data. ts_tls_pump_recv saw rc=-107, which is
      not -EAGAIN(-11), returned -1 → "recv failed: -1" in status bar.
    - FIX: call tcp_recv() FIRST to drain buffered data, THEN check
      connection state. Standard TCP half-close behavior — data may
      arrive before FIN is processed.
    - Also bumped SYS_RECV serial trace limit from 5 to 50.
    - Result: YouTube now receives 253KB (up from 138KB before error).

12. BR_SSL_CLOSED HANDLING — ts_http.h ts_http__recv()
    - BR_SSL_CLOSED with any error code now returns 0 (EOF) instead
      of -1 (fatal). Server may reset after sending all data.
    - Logs: TLS_CLOSED: err=N body=N for diagnosis.
    - YouTube triggers err=7 (BR_ERR_BAD_MAC) — truncated final TLS
      record when server closes connection. This is expected/harmless.
