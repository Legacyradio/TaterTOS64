fry812 — TaterSurf rule_count corruption investigation + CSS bounds guards
Date: 2026-04-04

== Summary ==
Investigated memory corruption where stylesheet.rule_count gets written to
128656 (0x1F690) during ts_doc_build(), even though CSS parser never runs.
Added bounds guards to all CSS rule iteration loops to prevent out-of-bounds
reads when rule_count is corrupted. Removed 5MB stack copy that caused the
original crash.

== Root Cause: Original Page Fault Crash ==
Page fault (vec=14, err=0x4) in ts_css_match_part() reading part->tag[0]
from an unmapped address. Root cause: 3 places in tatersurf.c copied
struct ts_stylesheet (~5 MB) to the stack:
  struct ts_stylesheet saved_ss = g_doc->stylesheet;
This overflowed the stack and corrupted memory.

Fix (from previous session, confirmed working):
- Added ts_doc_reinit() that resets render tree fields without touching
  the stylesheet or img_cache (ts_layout.h ~line 462)
- Replaced all 3 save/restore blocks in tatersurf.c with ts_doc_reinit()
- No more 5 MB stack allocations

== Memory Corruption: rule_count = 128656 ==
After fixing the crash, the browser renders but with wrong styling.
Investigation found stylesheet.rule_count is corrupted from 0 to 128656
during ts_doc_build().

Diagnostic trail:
1. DOC_INIT diagnostic: sizeof=6726856 rc_off=6642980 rc=0 bytes=00000000
   → ts_doc_init() correctly zeros entire struct, rule_count starts at 0
2. CSS_PARSE diagnostic: never fires
   → No <style> blocks parsed, no CSS rules added (Google uses inline styles)
3. PRE_BG diagnostic (inside html/body handler): rc=128656 nodes=4 boxes=0
   → By node 4, rule_count is already corrupted to 128656
4. BUILD_STACK: sp~0x7fffffff3e30 doc=0x100000000020
   → Stack at 0x7fffffff..., doc at 0x100000000..., no overlap
5. RC_HIT watchpoint (top of main loop): NEVER fires
   → rule_count is 0 at top of each loop iteration
   → Corruption occurs BETWEEN the loop-top check and the PRE_BG fprintf
   → Only pure read-only tag_name_eq comparisons happen between these points
   → No doc writes occur in that span

Struct layout (verified by diagnostic):
  doc base:          0x100000000020
  stylesheet.rules:  offset 1,477,924 (512 rules × 10,088 bytes each)
  stylesheet.rule_count: offset 6,642,980
  doc end:           offset 6,726,856 (0x10000066a4e8)

Key observations:
- Value 128656 = 0x1F690, grows to 128829 between PRE_BG calls
- User stack = 16 MB (4096 pages) — no stack overflow
- Heap allocation = 1643 pages (6,729,728 bytes) — fits the 6,726,856 struct
- No array overflow from nodes/text/links/forms/boxes can reach offset 6,642,980
- The corruption may be a compiler code-generation artifact or a subtle
  interaction with the bare-metal runtime; root cause NOT yet identified

== Fix: CSS Bounds Guards ==
Since root cause is unidentified, added defensive bounds guards to ALL loops
that iterate over stylesheet rules. This prevents the cascade of damage:
corrupted rule_count → read 128K garbage "rules" → apply random CSS → broken page.

Changed files:

ts_layout.h — ts_doc__match_apply_css() (~line 1284):
  - Was: for (i = 0; i < doc->stylesheet.rule_count; i++)
  - Now: cap rule_count to [0, TS_CSS_MAX_RULES], cap selector_count to
    [0, TS_CSS_MAX_SELECTORS], validate part_count <= TS_CSS_MAX_SELECTOR_PARTS

ts_layout.h — ts_doc__open_box() CSS matching (~line 1025):
  - Same bounds guards applied to rule iteration loop

ts_dom_bindings.h — getComputedStyle() rule matching (~line 1268):
  - Same bounds guards applied

Why: Even with corrupted rule_count=128656, the capped loops iterate over
at most 512 rules (all zero-initialized since CSS_PARSE never ran), so
no garbage CSS gets applied. This neutralizes the corruption's visible effect.
