§2 - PROBLEM #1: TATERTOS HAS NO USERLAND THREADING (yet)
=====================================================================
AK + LibCore call pthread_create / pthread_mutex / pthread_cond /
pthread_self / pthread_key (TLS). TaterTOS today is one-thread-per-
process. Plan v3 Phase 23 already specifies the missing primitives:

  SYS_THREAD_CREATE (70)
  SYS_THREAD_EXIT   (71)
  SYS_THREAD_JOIN   (72)

The kernel ALREADY HAS THE HARD PART:
  src/kernel/proc/sched.h — sched_block_futex, sched_wake_futex
                            (used today for inter-process wakeups,
                             generalizes 1:1 to intra-process wakeups)

Honest path forward (NO shortcut): bring Phase 23 thread syscalls
forward as a prerequisite. This is real, plan-conforming work, not
a corner cut. Concrete steps below in §4.

NOT acceptable:
  - Cooperative green threads (setjmp/longjmp) → not real preemption,
    will livelock LibWeb's idle/network threads.
  - Stub pthread_create to run synchronously → silently breaks
    LibWeb's image decoder, RequestServer, ThreadPool.
  - Single-thread Ladybird via -DAK_NO_THREADS → upstream has no
    such flag and adding one means modifying ~40 files in AK alone.

§3 - PROBLEM #2: EXCEPTIONS
=====================================================================
Ladybird builds with C++ exceptions enabled. TaterTOS userland has no
unwinder. Two real options (no shortcuts):

  (3a) Build Ladybird with -fno-exceptions and convert every `throw` to
       AK's existing ErrorOr<T> idiom. SerenityOS shipped this way for
       7 years against the same codebase. The throw sites are shallow
       and ErrorOr<T> is already pervasive. Estimated audit: ~30 files.
       This is a real engineering exercise, not a shortcut, IF we
       audit and convert each throw. Trackable.

  (3b) Port libunwind (LLVM, Apache 2.0 with LLVM exception) to
       TaterTOS userland. Real work, but adds a significant new
       runtime that nothing else in TaterTOS uses. ~5k LOC port.

Recommendation: 3a. Reason: 3b adds permanent surface area for one
consumer; 3a converts upstream Ladybird to match TaterTOS error
conventions (ErrorOr is more rigorous than exceptions anyway). The
"audit and convert" is the work; it is not minimal.

§5 - PROBLEM #3: C++ STDLIB GLUE FOR AK'S 5 HEADERS
=====================================================================
AK includes <initializer_list> <iterator> <memory> <new> <utility>.
The AUR x86_64-elf-g++ 15.2.0 ships libstdc++ headers that work
freestanding for these 5 (verified by inspection: they're template
glue only, no runtime). They are LGPL-with-runtime-exception, which
is a permissive license for header-only use.

OPTION A: use the cross-compiler's own libstdc++ headers as-is. This
is what every other freestanding C++ kernel does (incl. SerenityOS).
Risk: some versions pull in <bits/c++config.h> which #includes
<features.h> (glibc-specific). If that breaks, we ship a 10-file
shim of just these 5 headers (we already have a model for this:
src/user/apps/openh264/shim/*).

OPTION B: vendor a tiny libc++-style replacement for these 5 headers
under src/user/apps/ladybird/shim/cxx/. ~500 LOC total. Pure templates.

Recommendation: try A first (fastest validation that nothing breaks);
fall back to B if A pulls in glibc paths. NEEDS ZACK'S CALL.
