THREADING SEMANTICS — what the userland shim must respect
=====================================================================
1. The thread group leader (pid == tgid) is the original process. It
   CANNOT be exited via SYS_THREAD_EXIT; trying returns -EINVAL.
   pthread_exit() called from main thread must therefore translate to
   process_exit_group (i.e. SYS_EXIT, not SYS_THREAD_EXIT).
2. Only siblings within the same tgid can join each other. Cross-
   group join returns -EINVAL. pthread shim must surface this as
   EINVAL or document it as a TaterTOS-specific constraint.
3. Threads share fd table, VM regions, heap, args_buf, and outbuf/
   inbuf via shared->. This matches POSIX.
4. Threads do NOT share kernel stacks or register contexts (each has
   its own kernel_stack_top, user_rsp, user_rip, user_fs_base).
5. user_fs_base on a new thread is initialized to 0 (process.c:297).
   pthread_setspecific / TLS will need the shim to set this
   per-thread (likely via SYS_SET_FS_BASE or similar — verify in
   syscall.c during fry819 next investigation).
6. Stack allocation: process_create_user_thread takes a
   user_rsp argument from the caller — it does NOT allocate the
   user stack. The pthread shim must mmap a stack region and pass
   the top of it as user_rsp. Stack size policy lives in userland.
