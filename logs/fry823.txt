THE RUST EVOLUTION
=====================================================================
Modern Ladybird (2026 main) has migrated chunks of LibUnicode and
LibRegex to Rust crates with C++ FFI bindings. This is a recent
upstream evolution. Building these on TaterTOS would require:
  1. A Rust toolchain that targets x86_64-unknown-tatertos-gnu
     (does not exist; building one is multi-month).
  2. cbindgen to generate the C++ FFI headers.
  3. Cargo + the Rust standard library ported to TaterTOS userland.

This is not happening in Chunk 2. Possibly never. Two real long-term
options for the Rust pieces:
  - Pin Ladybird to a pre-Rust commit (probably late 2025 / early
    2026) that predates the Rust adoption.
  - Reimplement the Rust functionality natively in C++ for TaterTOS.

Decision deferred to Chunk 4 (when LibWeb actually consumes Unicode
+ Regex). For Chunk 2, we sidestep entirely (see THE FINDING below).

==========================================================================
THE FINDING — LibCore's actual coupling to sister libs
==========================================================================
CMake says LibCore PRIVATE-links LibUnicode + LibURL + LibTextCodec.
But what does LibCore's source code ACTUALLY reference from those
libraries?

Grep result over ~/ladybird-upstream/Libraries/LibCore/*.cpp:
  Files that include any sister-lib header: 1
    Libraries/LibCore/MimeData.cpp

  Sister-lib headers actually included: 1
    #include <LibURL/Parser.h>

That's it. LibUnicode and LibTextCodec are PRIVATE link deps that no
file in LibCore actually #includes or references at the source level.
They're transitive deps pulled in by LibURL/Parser.h's own includes.

Implication: if MimeData.cpp doesn't need LibURL, then LibCore.a
doesn't need ANY of the three sister libs to compile or link.

==========================================================================
WHAT MimeData.cpp USES LibURL FOR
==========================================================================
Reading Libraries/LibCore/MimeData.cpp:
  Single use site: URL parsing inside guess-mime-from-URL helpers.
  This is browser-shaped functionality (extension-from-URL → MIME
  type), not core LibCore functionality (which is event loops, file
  I/O, sockets, processes).

  When LibWeb in Chunk 4 lands, LibURL gets ported properly and
  MimeData.cpp re-enables its URL path.

THREE OPTIONS FOR CHUNK 2 (need Zack call)
==========================================================================

U1. CARVE OUT MimeData.cpp from Chunk 2's LibCore build.
    LibCore.a builds without MimeData. Other LibCore consumers don't
    need MIME guessing in Chunk 2 (LibWeb lands in Chunk 4 and brings
    its own MIME machinery). LibUnicode/LibURL/LibTextCodec/LibRegex
    are entirely deferred to Chunk 4.

    Pros:
      - Chunk 2 collapses to AK + LibCore (minus MimeData) + 3 vendored
        header-only libs. Genuinely 2 libraries, not 5.
      - No ICU. No Rust. No public_suffix codegen. No
        libtextcodec_generators codegen.
      - Chunk 2 exit criterion is small enough to actually hit.
    Cons:
      - LibCore diverges from upstream by 1 file. Marked divergence,
        easy to upstream or revert later.
    Recommendation: U1.

U2. STUB MimeData.cpp's URL path with TaterTOS-native ts_url.h.
    Patch MimeData.cpp's #include of LibURL/Parser.h with a thin
    bridge that calls TaterTOS's existing ts_url.h (already in
    src/user/apps/ — written for TaterSurf). Slightly bigger
    divergence than U1 but keeps full LibCore functionality.

    Pros:
      - Full LibCore parity with upstream.
    Cons:
      - More divergence to maintain. ts_url.h's API doesn't match
        LibURL's API; the bridge is real glue code.
      - Not road-building, it's bridge-building.

U3. PORT A SLIM LibURL FOR CHUNK 2.
    Pull in LibURL but stub out its LibUnicode/LibTextCodec/LibRegex
    transitive deps. LibURL/Parser.cpp uses LibUnicode for IDNA
    (international domain names) — stub IDNA to ASCII-only for v1.
    LibURL/Pattern/* uses LibRegex — drop the Pattern subdir entirely
    (only used by URLPattern API, not vanilla URL parsing).

    Pros:
      - Real LibURL surface for downstream consumers.
    Cons:
      - More work in Chunk 2 (~15 LibURL files + IDNA stub).
      - Pulls forward Chunk 4 work without the Chunk 4 framing.
    Recommendation: NO. This is the "build the road" trap in reverse —
    we'd end up half-porting LibURL twice, once now and once properly
    in Chunk 4.
