DESIGN DECISIONS
=====================================================================

R2-D1  Modeled on x86_64-unknown-linux-musl, NOT x86_64-unknown-none.
  Reason: x86_64-unknown-none is no_std-only (no thread support,
  kernel code-model, no syscall ABI). Ladybird's Rust crates use
  std (Vec, String, HashMap, threads) so we need a target where
  rustc believes std is buildable. The musl analog matches our
  posture: have a libc, have threads, have processes, statically
  linked.

R2-D2  os = "tatertos".
  This is the cfg gate Bucket C crates (libc, getrandom, etc.)
  will key off via `#[cfg(target_os = "tatertos")]`. fry844
  classified ~7 crates as needing this branch.

R2-D3  target-family = ["unix"].
  Allows std::os::unix::* code paths to compile. TaterTOS's POSIX
  surface is unix-flavored (we have everything from `unistd.h`,
  `pthread.h`, `sys/socket.h`, etc.). Crates that gate on
  target_family will see us as Unix-y by default — fewer per-crate
  patches.

R2-D4  panic-strategy = "abort".
  Matches Cargo.toml workspace setting. Matches our C++
  -fno-exceptions posture from Chunk 2 (D2 in fry817). No
  unwinder needed.

R2-D5  executables = false; dynamic-linking = false.
  Rust crates ship as staticlib (.a) only. Final TaterTOS .fry
  binaries are produced by our Make/LD pipeline that already
  links AK.a + LibCore.a — Rust's .a files just join that link
  line.

R2-D6  features = "+sse,+sse2".
  TaterTOS x86_64 always has SSE/SSE2 (Coffee Lake target, Dell
  7530). We avoid +avx as a baseline because higher SIMD is
  enabled per-translation-unit by simdutf's runtime kernel
  selection — promoting the whole binary to AVX would break
  early-boot CPUs that haven't initialised XCR0.

R2-D7  static-position-independent-executables = false,
       position-independent-executables = false.
  No PIE on TaterTOS yet — kernel ELF loader doesn't relocate
  user binaries (matches the existing C/C++ user app build).

R2-D8  code-model = "small".
  Userland binaries fit in the 2GB code model. Kernel uses
  "kernel" code-model; userland = "small" (default).

R2-D9  panic_handler is the user's responsibility.
  Vendored crates that include build.rs + main.rs handle their
  own no_std panic_handler. std-using crates inherit std's
  panic infrastructure (which we'll wire up in Phase R-6).
