THIRD-PARTY DEPENDENCY: rust-src ARCH PACKAGE
=====================================================================
Installed: extra/rust-src 1:1.95.0-1 (atomic version match with rustc)
On disk:   /usr/lib/rustlib/src/rust/library/

Why install vs. keep-our-extracted-tarball:
  - Cargo's `-Zbuild-std` looks at $(rustc --print sysroot)/lib/
    rustlib/src/rust/library/ by default. With Arch's rust-src
    package, that path is populated automatically.
  - Pacman ATOMICALLY version-locks rust-src to the same release
    as rustc. If Arch ever bumps rust to 1.96, both packages move
    together. Our in-tree extract would silently become a
    version-mismatch trap.
  - One supported source-of-truth is better than two copies that
    can drift.

Removed: src/user/apps/ladybird-rust-vendor/rust-src/ (the 51 MB
fry846-extracted tarball subtree). The Arch package supersedes it.

CARGO CONFIG CHANGES (.cargo/config.toml)
=====================================================================
Added two stanzas to src/user/apps/ladybird-rust-vendor/.cargo/config.toml:

  [build]
  target = "x86_64-unknown-tatertos"
  rustflags = ["-Zunstable-options"]    ← NEW, fry847

  [unstable]                            ← NEW, fry847
  build-std = ["core", "alloc"]

The rustflags = ["-Zunstable-options"] line is REQUIRED. Without it,
rustc rejects the custom JSON target with:
  error: error loading target specification: custom targets are
  unstable and require `-Zunstable-options`

Cargo's --target=<json> path triggers this in rustc-the-binary
on its host info probe; cargo passing the JSON alone is not enough.

The [unstable] block is parsed by stable cargo only with
RUSTC_BOOTSTRAP=1 in env. We invoke as:

  RUSTC_BOOTSTRAP=1 \
  RUST_TARGET_PATH=/home/legacyindieradio/TaterTOS64/src/user/apps/ladybird-rust-vendor \
    cargo build --release

(RUST_TARGET_PATH lets the [build] target setting reference the
custom target by bare name "x86_64-unknown-tatertos" rather than
a JSON path — cargo searches RUST_TARGET_PATH for <name>.json.)

SMOKE CRATE
=====================================================================
NEW: src/user/apps/ladybird-rust-vendor/smoke/Cargo.toml (~10 LOC)
NEW: src/user/apps/ladybird-rust-vendor/smoke/src/lib.rs  (~20 LOC)

  #![no_std]
  #[panic_handler] fn panic(_: &PanicInfo) -> ! { loop {} }
  #[unsafe(no_mangle)]
  pub extern "C" fn tater_rust_smoke(x: u64) -> u64 {
      x.wrapping_mul(3).wrapping_add(7)
  }

(`#[unsafe(no_mangle)]` is the new edition 2024 syntax. Plain
`#[no_mangle]` warns; `unsafe(no_mangle)` is the canonical form.)

profile sets `panic = "abort"` matching our target spec's
`"panic-strategy": "abort"`. No unwinder linked, no eh_personality
needed.

Cargo.toml uses [lib] crate-type = ["staticlib"] so the output
is a self-contained .a that the C++ link line can ingest like any
other libfoo.a.
