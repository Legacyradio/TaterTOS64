POSIX / LIBC / C++ SHIM AUDIT
=====================================================================
P1. Large POSIX surface was added for Ladybird.
    Added headers include arpa/inet.h, dirent.h, fcntl.h, math.h,
    netdb.h, poll.h, pthread.h, pwd.h, sched.h, shadow.h, signal.h,
    spawn.h, stdio.h, stdlib.h, string.h, strings.h, sys/ioctl.h,
    sys/mman.h, sys/resource.h, sys/select.h, sys/socket.h,
    sys/stat.h, sys/time.h, sys/types.h, sys/un.h, sys/wait.h,
    termios.h, time.h, unistd.h, utime.h.

P2. Many constants intentionally match Linux.
    Examples in errno.h, fcntl.h, poll.h, signal.h, sys/ioctl.h,
    sys/socket.h, sys/stat.h, netinet/in.h. This may be a pragmatic
    POSIX compatibility choice, but it must not turn into "TaterTOS is
    Linux ABI." Any such constants should be documented as numeric ABI
    compatibility only.

P3. Several declarations exist without real implementation yet.
    unistd.h declares link/linkat/unlinkat/mknod/mkfifo/pipe2/pause/
    nice/acct/domain/session/process-group/tty/temp-file/timestamp/
    sendfile/eventfd/timerfd/signalfd and more. Some may be backed in
    posix.c, some are likely still unresolved until an executable links.
    This means "LibCore.a built" is weaker than "runtime surface works."

P4. cxx_runtime.cpp contains unsafe/incomplete C++ runtime behavior.
    - throwing operator new returns nullptr even though standard throwing
      new should not return null.
    - __cxa_guard_acquire spins instead of using futex despite comment.
    - exception/unwind symbols are abort/no-op stubs.
    This is enough to link a smoke test, not enough to declare C++
    runtime complete for browser work.

P5. pthread implementation is real in parts but incomplete.
    pthread_create/join/mutex/cond/rwlock/once/TLS exist, but detach,
    names, TLS destructors, key deletion, and attr handling contain
    no-op/stub behavior. This needs feature-by-feature validation
    against Ladybird's actual thread usage.

P6. errno is made TLS-backed. This is good, but it allocates per thread
    and has fallback behavior if TLS init or malloc fails. Need stress
    testing with multiple Ladybird threads before relying on it.

P7. Old OpenH264 shim files were modified as collateral.
    assert.h and OpenH264 shim headers got C++ linkage fixes and
    ts_h264_wrap.cpp include wrapping changed. These are browser-adjacent
    because old TaterSurf uses OpenH264, but they are not Ladybird.

NETWORK AUDIT
=====================================================================
N1. netcore.c has a QEMU-SLIRP-specific comment and behavior:
    "QEMU SLIRP user-mode networking doesn't send ARP replies"
    It opportunistically learns the gateway MAC from incoming IP frames.

N2. This may be useful, but it is currently not isolated behind explicit
    runtime detection. planv3 Rule 15 requires hardware-agnostic behavior
    and isolation of quirks. The comment frames the fix around QEMU, so
    it should be reworked or re-justified before being kept.

N3. netcore_attach_e1000 still logs "(QEMU)" for e1000. That predates
    this audit, but it shows QEMU assumptions remain in network naming.
