Progress Made This Session
Two bugs were diagnosed and fixed:

1) pd_high mapping bug (fry242) — FIXED PRIOR SESSION, CONFIRMED WORKING
   - pd_high[0] was mapping virt 0xffffffff80000000 → phys 0x000000 instead of phys 0x200000.
   - Fixed: changed `(m * 0x200000)` to `((m+1) * 0x200000)` in efi_boot.asm pd_high table.
   - Confirmed: debugcon shows "kgt" = gdt_init, idt_init, tss_init all completed. Kernel now runs.

2) Windows ABI / System V ABI calling convention mismatch (fry243) — FIXED THIS SESSION
   - Root cause: efi_loader.c is compiled by build_iso.sh with x86_64-w64-mingw32-gcc
     (Windows/ms_abi). Windows ABI passes the first argument in RCX. The kernel
     (_fry_start, System V ABI) expects the handoff pointer in RDI.
   - efi_boot_start had no ABI translation — it called _fry_start directly without
     moving RCX to RDI. Result: _fry_start received whatever garbage UEFI left in RDI.
   - Evidence: at the pmm_init fault, R8 (= handoff->mmap_base) = 0x118 (280 — wrong),
     R9 (= handoff->mmap_size) = 0xafafafaf6c617470 (heap garbage). The pmm_init first
     loop ran ~44 million iterations with this garbage until the descriptor pointer RSI
     reached 0x100000018 (4GB+24 = unmapped), causing #PF at CR2=0x100000030.
   - Fix: added `mov rdi, rcx` in efi_boot_start before `call rax` (_fry_start).
   - Verified in binary: offset 0x10001f = `48 89 cf` = mov %rcx,%rdi ✓
