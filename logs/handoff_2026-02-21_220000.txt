STEP 2 — Fix SMP trampoline low memory
────────────────────────────────────────────────────────────────────────────────
Why second: Once kprint is readable we can see real SMP output. Two CPUs are
detected but APs never start because trampoline code has nowhere to live.

Plan:
  - Read src/drivers/smp/smp.c. Find where "no low memory for trampoline" fires.
  - Trampoline needs a 4KB-aligned page below 0x100000 (e.g. 0x8000–0x8FFF).
  - The region 0x8000–0x9FFF is conventional free RAM in QEMU — not used by
    BIOS IVT (0x0–0x3FF), BDA (0x400–0x4FF), or our boot code (0x100000+).
  - Strategy: reserve that page in pmm_init (mark as used so PMM won't hand it
    out), then in smp_init write the trampoline there at a known fixed address.
  - Alternatively: allocate from PMM early before pmm_init marks everything used.
  - Copy 16-bit AP startup stub to chosen physical address.
  - Send INIT/SIPI/SIPI to each AP LAPIC.
  - Confirm APs check in (e.g. atomic counter incremented by each AP).
  - Log: fry258.txt

Success criteria:
  Serial shows "SMP: AP 1 online" (or equivalent) — no "no low memory" message.
  debugcon shows no triple faults from APs.
