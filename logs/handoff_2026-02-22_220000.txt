ROOT CAUSE FOUND (fry310 needed)
---------------------------------
Disassembled kernel.elf. GCC inlined context_switch() into sched_tick() with -O2.

context_switch has inline asm that does:
  16 pushes → save RSP to from->saved_rsp → load to->saved_rsp → 16 pops → ret

The `ret` is designed to "return" to whatever is at position 16 above the loaded RSP:
  - For a FRESH process: setup_initial_stack put process_start there → works
  - For RESUMING a process: expects the return address from `call context_switch`
    to be there, so ret returns into sched_tick at the instruction after the call

BUG: When GCC INLINES context_switch into sched_tick, there is NO `call` instruction.
No call → no return address pushed onto the stack before the 16 asm pushes.
So from->saved_rsp+128 is NOT a return address — it's a LOCAL VARIABLE inside
sched_tick's stack frame (the slot at rbp-0x30 in sched_tick's frame).

Timeline of failure:
  1. FIRST timer interrupt fires on main boot thread (ring 0, on kernel_stack)
     sched_tick → inlined context_switch(idle, shell):
       - 16 pushes on main kernel_stack
       - idle->saved_rsp saved (now points into main kernel_stack)
       - shell's initial stack loaded, 16 pops, ret → process_start → iretq → shell ✓

  2. SECOND timer interrupt fires while shell runs (ring 3, on shell kstack):
     sched_tick → inlined context_switch(shell, idle):
       - 16 pushes on shell kstack (shell->saved_rsp set)
       - Load idle->saved_rsp (points into main kernel_stack's first sched_tick frame)
       - 16 pops from main kernel_stack → RSP ends up at sched_tick's local var area
       - `ret` pops a LOCAL VARIABLE SLOT, not a return address
       - That slot contains 0xFFFFFFFF8023FF7C (a kernel_stack address produced by
         some pointer arithmetic GCC spilled there during first sched_tick execution)
       - CPU jumps to 0x8023FF7C (data in kernel_stack) → invalid bytes → #UD → loop

Confirmed by disassembly at sched_tick+0x188..+0x1c9 (ffffffff80014140..80014179):
  pushf / push rax..r15 / mov rsp, 0x10(rdx) / mov 0.10(r12), rsp
  pop r15..rax / popf / ret
  (context_switch inlined directly into sched_tick, no `call` instruction)

THE FIX (fry310)
-----------------
File: src/kernel/proc/sched.c

Change context_switch declaration from:
  static void context_switch(struct fry_process *from, struct fry_process *to) {

To:
  __attribute__((noinline))
  static void context_switch(struct fry_process *from, struct fry_process *to) {

With noinline, sched_tick emits `call context_switch`. The call pushes a return
address onto the stack. Inside context_switch:
  call pushes RETURN_ADDR → RSP = Q - 8
  16 asm pushes → RSP = Q - 136
  from->saved_rsp = Q - 136

When restoring from's context later:
  load RSP = Q - 136
  16 pops → RSP = Q - 8
  ret → pops RETURN_ADDR (= instruction in sched_tick after `call context_switch`) ✓

For a FRESH process (setup_initial_stack):
  heap kstack top - 17 items. saved_rsp = kstack_top - 136.
  After 16 pops: RSP = kstack_top - 8 = process_start slot.
  ret → process_start ✓

IMPORTANT: With noinline + -O2, GCC should NOT add callee-saved register pushes
to context_switch (no function calls inside, simple control flow). VERIFY the
disassembly of context_switch after build to confirm no extra prologue pushes.
If GCC adds any (push rbx, push r12 etc.), setup_initial_stack needs adjustment
or __attribute__((naked)) + full asm must be used.
