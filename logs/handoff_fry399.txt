========================================================================
ROOT CAUSE FOUND AND FIXED
========================================================================
PID_DMI was set to 0x88 (wrong — undefined sideband port on Intel 300-series
PCH) instead of 0xEF (correct DMI sideband port on ALL Intel PCHs from
Skylake through Raptor Lake).

This means for the last 8 fry sessions (fry391-fry398), the PCR[DMI] sideband
mirror write was going to physical address 0xFD882774 (unknown PID 0x88 space)
instead of 0xFDEF2774 (actual DMI LPCIOE register). The EC I/O decode bit
was never set in the hardware decode logic, even though PCI config at offset
0x82 showed it as enabled (0x3F0F).

Reference: coreboot src/soc/intel/cannonlake/include/soc/pcr_ids.h defines
PID_DMI = 0xEF for all Skylake+ PCHs.

========================================================================
SMP FIX DETAILS
========================================================================
alloc_stack() now uses pmm_alloc_pages() + vmm_phys_to_virt() instead of
kmalloc(). The bump-allocator heap was only ~1MB and couldn't free, so
7072 AML namespace nodes + NVMe/VMD init consumed it before SMP could
allocate 192KB for AP stacks. PMM has abundant physical pages.
